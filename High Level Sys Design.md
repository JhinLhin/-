# ByteByteGo

## 05 Design A Rate Limiter

> ------
>
> ## 第一步：澄清需求（开场必做！）
>
> > 面试一开始，先问问题，别急着设计。这显示你思路清晰。
>
> 你说：
>
> > "在开始设计之前，我想先确认几个问题——"
>
> **要问的问题：**
>
> | 问题                           | 为什么问           |
>| ------------------------------ | ------------------ |
> | 是客户端限流还是服务端限流？   | 方向完全不同       |
>| 按什么限流？IP？用户ID？       | 影响数据结构设计   |
> | 系统规模多大？                 | 决定是否需要分布式 |
> | 需要在分布式环境下工作吗？     | 影响架构复杂度     |
> | 限流器是独立服务还是嵌入代码？ | 部署方式不同       |
> | 被限流的用户需要收到通知吗？   | 影响响应设计       |
> 
> **得到的答案（标准版）：**
> 
> - 服务端限流
>- 支持灵活规则（IP、用户ID都要支持）
> - 大规模系统，需要分布式
>- 设计决定由你来做
> - 需要通知用户
> 
> ------
> 
> ## 第二步：说出核心需求
>
> > 把需求总结出来，让面试官点头确认。
>
> **功能需求：**
>
> - 准确限制超额请求
>- 用户被限流时要收到清晰提示
> 
>**非功能需求：**
> 
> - 低延迟（不能拖慢正常请求）
>- 内存占用小
> - 支持分布式（多台服务器共享限流状态）
>- 高容错（限流器挂了不能影响主系统）
> 
> ------
> 
> ## 第三步：高层设计
>
> ### 3.1 先说"放在哪里"
>
> > 这是第一个设计决策，要主动提出来。
>
> **三个选项：**
>
> **① 放在客户端** ❌
>
> - 客户端代码可以被恶意用户篡改，不安全
>- 我们控制不了客户端实现
> 
>**② 放在服务器代码里** ✅ 可以
> 
> - 完全控制算法
>- 但每个服务都要自己实现，重复工作多
> 
>**③ 放在中间件（API Gateway）** ✅ 推荐
> 
> - 独立的一层，客户端请求先经过它再到服务器
>- 还可以顺便做：SSL、身份验证、IP白名单
> - 微服务架构下最常用
>
> > **面试金句：** "我推荐将限流器实现为中间件，这样可以与业务逻辑解耦，并且在微服务架构中可以统一管理。"
> 
> ------
>
> ### 3.2 介绍算法（重要！）
>
> > 说"我知道有几种算法，让我分析一下各自的优缺点"
>
> #### 算法一：Token Bucket（令牌桶）⭐ 最常用
>
> **直觉理解：** 想象一个桶，系统每秒往里放N个令牌。每来一个请求，取一个令牌。桶空了就拒绝请求。
>
> **关键参数：**
>
> - 桶容量（burst size）：最多能存多少令牌
>- 补充速率：每秒放多少令牌
> 
>**特点：** 允许短时间突发流量（桶里有存货就能用）
> 
> ✅ 优点：实现简单、内存效率高、允许突发
> ❌ 缺点：两个参数难以调优
> 
>> **使用公司：** Amazon、Stripe
> 
> ------
>
> #### 算法二：Leaking Bucket（漏桶）
>
> **直觉理解：** 请求先进队列排队，系统以固定速率处理队列。队列满了就丢弃新请求。
>
> **特点：** 输出速率固定，像水从桶底漏出去一样均匀
>
> ✅ 优点：输出稳定，适合需要均匀处理的场景
> ❌ 缺点：突发流量会塞满队列，新请求会被丢弃
> 
>> **使用公司：** Shopify
> 
> ------
>
> #### 算法三：Fixed Window Counter（固定窗口计数）
>
> **直觉理解：** 把时间切成固定格子（比如每分钟一格），每格独立计数。超过上限就拒绝。
>
> **致命缺陷：** 窗口边界问题——如果允许每分钟5个请求，用户可以在00:59发5个，01:01再发5个，实际上1秒内发了10个！
>
> ✅ 优点：简单、内存高效
> ❌ 缺点：边界处可能放过双倍流量
> 
>------
> 
> #### 算法四：Sliding Window Log（滑动窗口日志）
>
> **直觉理解：** 记录每个请求的精确时间戳。每次新请求来时，只统计"最近1分钟"内的请求数量（滑动的窗口，不是固定格子）。
>
> **流程：**
>
> 1. 删除1分钟前的旧时间戳
>2. 添加当前时间戳
> 3. 如果日志大小 ≤ 上限，放行；否则拒绝
>
> ✅ 优点：非常精确，任意滑动窗口内都不超额
>  ❌ 缺点：**内存消耗大**（被拒绝的请求时间戳也要存）
> 
>------
> 
> #### 算法五：Sliding Window Counter（滑动窗口计数）⭐ 推荐
>
> **直觉理解：** 结合了固定窗口的低内存 + 滑动窗口的精确性。
>
> **公式：**
>
> ```
>当前窗口请求数 + 上一窗口请求数 × 当前时间点在本窗口的百分比
> ```
>
> **例子：** 限制每分钟7个请求。上一分钟5个，当前分钟3个，现在处于当前分钟的30%位置：
> 
> ```
>3 + 5 × 70% = 6.5 → 取整为6，未超限，放行
> ```
>
> ✅ 优点：平滑处理突发、内存高效
>  ❌ 缺点：是近似值，极端情况下有误差（但Cloudflare测试显示误差率仅0.003%）
> 
>------
> 
> #### 算法对比表
>
> | 算法                   | 内存   | 精确度   | 允许突发 | 推荐场景       |
>| ---------------------- | ------ | -------- | -------- | -------------- |
> | Token Bucket           | 低     | 中       | ✅        | 通用，最常用   |
>| Leaking Bucket         | 低     | 中       | ❌        | 需要稳定输出   |
> | Fixed Window           | 最低   | 低       | 看情况   | 简单场景       |
> | Sliding Window Log     | **高** | **最高** | ❌        | 精确度要求极高 |
> | Sliding Window Counter | 低     | 高       | 部分     | 分布式推荐     |
> 
> ------
> 
> ### 3.3 高层架构图
>
> > 说完算法后，画出基本架构
>
> **核心问题：计数器存哪里？**
>
> - ❌ 数据库（MySQL）：磁盘读写太慢
>- ✅ **Redis（内存缓存）**：极快，支持原子操作
> 
>**Redis的两个关键命令：**
> 
> - `INCR`：计数器 +1
>- `EXPIRE`：设置过期时间（窗口结束自动清零）
> 
>**基本流程：**
> 
> ```
>客户端 → 限流中间件 → (查询Redis计数器) → API服务器
>                           ↑
>                         Redis
> ```
> 
> 1. 客户端发请求到限流中间件
> 2. 中间件去Redis查对应的计数器
> 3. 未超限 → 转发给API服务器，同时Redis计数+1
>4. 已超限 → 返回 **HTTP 429** (Too Many Requests)
> 
> ------
> 
> ## 第四步：深入设计
>
> > 面试官这时会追问细节，或者你主动提出
>
> ### 4.1 限流规则存哪里？
>
> 规则写在**配置文件**里（YAML格式），存在磁盘上：
>
> ```yaml
>domain: auth
> descriptors:
>  - key: auth_type
>     value: login
>     rate_limit:
>       unit: minute
>       requests_per_unit: 5  # 每分钟最多登录5次
> ```
> 
> **工作流程：** Workers（后台程序）定期从磁盘读取规则 → 存入**Cache（缓存）**→ 限流中间件从缓存读（速度快）
> 
> ------
>
> ### 4.2 如何通知用户被限流？
>
> **HTTP响应头（Response Headers）：**
>
> | Header                    | 含义                 |
>| ------------------------- | -------------------- |
> | `X-Ratelimit-Remaining`   | 当前窗口还剩几次请求 |
>| `X-Ratelimit-Limit`       | 每个窗口最多几次     |
> | `X-Ratelimit-Retry-After` | 多少秒后可以重试     |
> 
> 被限流时返回：**429 Too Many Requests** + `X-Ratelimit-Retry-After`
> 
> ------
>
> ### 4.3 分布式环境下的挑战
>
> > 这是面试最爱考的部分！主动提出来显得很厉害
>
> **挑战一：Race Condition（竞争条件）**
>
> **问题：** 两个服务器同时读到Redis计数器=3，都以为自己是第4个请求，都写回4，实际应该是5！
>
> **解决方案：**
>
> - 使用 **Lua Script**（Redis原生支持的脚本，读+写是原子操作，不会被打断）
>- 使用 **Redis的sorted sets**（有序集合，支持原子操作）
> 
>> **面试金句：** "我们可以用Lua Script来保证读-检查-写的原子性，避免竞争条件。"
> 
> ------
>
> **挑战二：Synchronization（同步问题）**
>
> **问题：** 多台限流服务器，每台都有自己的计数。用户今天请求了限流器1，明天请求了限流器2，两台不共享数据，限流形同虚设！
>
> **错误方案：** Sticky Session（让同一用户永远打到同一台服务器）→ 不灵活、不可扩展
>
> **正确方案：** 所有限流器共享同一个 **集中式Redis**
>
> ```
>限流器1 ──→ Redis（共享）
> 限流器2 ──→       ↑
>```
> 
> ------
> 
> ### 4.4 性能优化
>
> **优化一：多数据中心 + 边缘节点**
>
> - 用户请求路由到**最近的边缘服务器**，减少延迟
>- Cloudflare在全球有194个边缘节点
> 
>**优化二：最终一致性（Eventual Consistency）**
> 
> - 多个Redis节点之间不需要强一致，允许短暂数据差异
>- 牺牲少量精确度，换取极大的性能提升
> 
>------
> 
> ### 4.5 完整详细架构
>
> ```
>客户端
>   ↓
>限流中间件
>   ├── 读取缓存中的规则
>   ├── 查询Redis计数器
>   │
>   ├── 未超限 → API服务器（处理请求）
>   │              → 更新Redis计数器
>   │
>   └── 已超限 → 返回429
>                ↓ (可选)
>             消息队列（稍后重试）
> 
> 后台：Workers定期从磁盘读规则 → 更新Cache
> ```
> 
> ------
> 
> ## 第五步：监控
>
> > 设计完了要说"我们还需要监控"，显示你考虑周全
>
> - 监控**规则是否有效**：如果太多合法请求被拒，规则太严了
>- 监控**算法是否有效**：如果突发流量时系统崩溃，可能需要换Token Bucket
> - 收集被限流请求的统计数据，定期调整参数
>

## 06 Design Consistent Hashing

> ------
>
> ## 🧠 先建立直觉：从"问题"开始
>
> 在讲一致性哈希之前，你必须先理解它解决的是什么问题。面试里永远先说问题，再说解法。
>
> ------
>
> ## 第一步：面试官问"你怎么把请求分发到多台服务器？"
>
> ### 你的回答起点：普通取模哈希
>
> 想象你有 4 台缓存服务器（server 0, 1, 2, 3），用户请求来了，你需要决定去哪台服务器找数据。
>
> 最简单的方法：
>
> ```
>服务器编号 = hash(key) % 服务器总数N
> ```
> 
> **举个例子：**
>
> - key0 的 hash 是 18358617，18358617 % 4 = **1** → 去 server 1 找
>- key1 的 hash 是 26143584，26143584 % 4 = **0** → 去 server 0 找
> 
> 这个方法很简单，而且在服务器数量**固定不变**的时候运行得很好。
>
> ------
>
> ## 第二步：主动说出"但是有个严重问题"
>
> 这是面试的关键时刻——你要主动暴露问题，显示你理解深度。
>
> **问题：如果有一台服务器挂了怎么办？**
>
> 假设 server 1 挂了，现在只剩 3 台服务器（N 从 4 变成 3）。
>
> 同样的 key，hash 值没变，但是：
>
> ```
>hash(key0) % 3 = 0   （原来是 1）
> hash(key1) % 3 = 0   （原来是 0，碰巧没变）
> hash(key2) % 3 = 1   （原来是 2）
> ...
> ```
> 
> **几乎所有 key 都映射到了不同的服务器上！**
>
> 这意味着什么？
>
> > 所有客户端都去"错误"的服务器找数据，缓存全部失效，所有请求打穿到数据库，数据库直接崩溃。这叫 **缓存雪崩（Cache Avalanche）**。
>
> **同理，如果你增加一台新服务器（N 从 4 变成 5），同样会导致大量 key 重新分配，同样的灾难。**
>
> ------
>
> ## 第三步："所以我们需要一致性哈希"
>
> > "为了解决这个问题，我会使用一致性哈希。它的核心思想是：无论增减服务器，只需要重新分配 **k/n** 的数据（k 是总 key 数，n 是服务器数），而不是几乎全部。"
>
> ------
>
> ## 第四步：解释一致性哈希的原理
>
> ### 概念1：哈希环（Hash Ring）
>
> **什么是哈希环？**
>
> 普通哈希的输出是一条直线：0, 1, 2, 3, ... 到某个最大值（比如 SHA-1 是 0 到 2¹⁶⁰-1）。
>
> 现在，**把这条直线的头尾连起来，变成一个圆环**，这就是哈希环。
>
> 想象一个时钟表盘，12点的位置是 0，顺时针一圈转回来又是 0。这个环就是哈希空间。
>
> ------
>
> ### 概念2：把服务器放到环上
>
> 用**同一个哈希函数**，对每台服务器的 IP 或名字做哈希，把服务器映射到环上某个位置。
>
> ```
>hash("server0的IP") → 环上某个点
> hash("server1的IP") → 环上另一个点
> ...
> ```
> 
> 现在 4 台服务器散布在环上的 4 个不同位置。
>
> ------
>
> ### 概念3：把 key 也放到环上
>
> 同样用这个哈希函数，把数据的 key 也映射到环上。
>
> ```
>hash("key0") → 环上某个点
> hash("key1") → 环上另一个点
> ...
> ```
> 
> ------
>
> ### 概念4：查找规则——顺时针找最近的服务器
>
> **规则极其简单：**
>
> > 从 key 所在的位置，**顺时针走**，遇到的第一台服务器，就是这个 key 存储的地方。
>
> 例如：
>
> - key0 顺时针走，先碰到 server 0 → 存在 server 0
>- key1 顺时针走，先碰到 server 1 → 存在 server 1
> - key2 顺时针走，先碰到 server 2 → 存在 server 2
> 
> ------
>
> ### 概念5：为什么增删服务器影响小？
>
> **加入一台新服务器 server4：**
>
> server4 出现在环上某个位置，比如在 server3 和 server0 之间。
>
> 现在只有**原来属于 server0、但在 server4 和 server0 之间的 key** 需要转移给 server4。
>
> 其他所有 key 顺时针走，遇到的服务器没变，**完全不需要动**。
>
> **移除 server1：**
>
> 原来属于 server1 的 key，现在顺时针继续走会碰到 server2，所以只有这些 key 转移到 server2。其他 key 完全不受影响。
>
> > **面试金句：** "一致性哈希保证了增删节点时，平均只有 1/n 的数据需要迁移，而不是几乎全部。"
>
> ------
>
> ## 第五步：主动说出"基础方案有两个问题"
>
> 又一次展示深度！
>
> ### 问题1：数据分布不均匀（分区大小不一致）
>
> 服务器在环上的位置是由哈希函数决定的，可能出现这种情况：
>
> server0、server1、server2 三台服务器碰巧都挤在环的同一段，server3 占了半个环的空间。
>
> 结果：**大量 key 都落到 server3**，其他服务器闲置。这叫**热点问题（Hotspot）**。
>
> 而且，当你删除一台服务器，它的邻居服务器需要承接全部的 key，承受的压力可能是其他服务器的好几倍。
>
> ### 问题2：删除节点后分区大小剧烈变化
>
> 如果 server1 被删除，server2 需要承接 server1 的全部负载，可能是 server0 和 server3 的两倍甚至更多。负载不均衡。
>
> ------
>
> ## 第六步：解决方案——虚拟节点（Virtual Nodes）
>
> ### 什么是虚拟节点？
>
> **真实世界里，不是每台服务器在环上只放一个点，而是放很多个点。**
>
> 比如 server0 在环上放 100 个虚拟节点：s0_1, s0_2, s0_3, ..., s0_100
>
> server1 同样放 100 个虚拟节点：s1_1, s1_2, ..., s1_100
>
> 这 200 个虚拟节点**均匀散布**在整个环上（因为哈希函数的输出是均匀分布的）。
>
> ### 查找规则不变
>
> key 顺时针走，遇到第一个虚拟节点（不管是 s0 的还是 s1 的），就找到了对应的真实服务器。
>
> ### 为什么解决了问题？
>
> - **分布更均匀：** 100 个点散布在环上，比 1 个点受偶然性影响小得多。虚拟节点越多，分布越接近理论均值。研究表明：100个虚拟节点时标准差约为均值的10%；200个时约为5%。
>- **容错性更好：** 某台服务器挂掉，它的虚拟节点散布在环的各处，负载会均匀分散到**所有**其他服务器，而不是全压在一个邻居上。
> 
> ### 权重控制（加分项）
>
> 如果某台服务器性能更强（比如内存 2 倍），可以给它分配 **2 倍的虚拟节点数量**，自然承载 2 倍的数据。这是灵活控制负载的方式。
>
> ------
>
> ## 第七步：补充"如何找到受影响的 key"
>
> 面试官可能追问：新增/删除服务器后，怎么知道哪些 key 需要迁移？
>
> **加入 server4 时：** 从 server4 的位置**逆时针走**，直到遇到上一个服务器（比如 server3）。这段区间（s3 到 s4 之间）的所有 key，从 server0 迁移到 server4。
>
> **删除 server1 时：** 从 server1 的位置**逆时针走**，直到遇到 server0。s0 到 s1 之间的所有 key，从 server1 迁移到 server2。
>
> ------
>
> ## 🎯 完整面试流程脚本（按顺序说）
>
> ```
>1. "最简单的方法是取模哈希：serverIndex = hash(key) % N"
> 
> 2. "但这有个严重问题：服务器数量一旦变化，几乎所有 key 
>    都要重新映射，导致缓存雪崩。"
> 
> 3. "为了解决这个问题，我会用一致性哈希。"
> 
> 4. "核心思想是：把哈希空间首尾相连变成一个环，服务器和 
>    key 都映射到环上，key 顺时针找最近的服务器。"
> 
> 5. "这样增删服务器时，只有约 1/n 的 key 需要迁移。"
> 
> 6. "但基础方案有两个问题：分区大小不均匀、热点问题。"
> 
> 7. "解决方案是虚拟节点：每台服务器在环上放多个点，
>    让分布更均匀，还能通过虚拟节点数量控制权重。"
> ```
> 
> ------
>
> ## 📊 终极对比表
>
> | 对比项           | 普通取模哈希 | 一致性哈希（基础） | 一致性哈希（虚拟节点） |
>| ---------------- | ------------ | ------------------ | ---------------------- |
> | 增删节点时迁移量 | **几乎全部** | 约 1/n             | 约 1/n                 |
> | 负载均衡         | 均匀         | **可能不均**       | **很均匀**             |
> | 实现复杂度       | 极简         | 中等               | 较复杂                 |
> | 支持权重         | 否           | 否                 | **支持**               |
> | 实际生产使用     | 不推荐       | 不推荐             | **推荐**               |
> 
> ------
>
> ## 🏭 实际应用场景（加分项）
>
> 说完原理后可以提一下在哪用：
>
> - **Redis Cluster** — 用一致性哈希分片，16384 个哈希槽（槽本质上就是虚拟节点的变体）
>- **Cassandra、DynamoDB** — 数据在节点间分布
> - **CDN** — 决定某个资源缓存在哪个边缘节点
> - **负载均衡器** — 将同一用户的请求路由到同一台服务器（会话粘性）
> 
> ------
>
> ## 一句话记忆
>
> > **"环 + 顺时针 + 虚拟节点 = 增删服务器只动 1/n 数据，负载均匀分布。"**

## 07 Design A Key-value Store

> # 分布式 Key-Value Store 设计 — 系统设计面试完整答案
>
> ------
>
> ## 🧠 先建立直觉：这道题在问什么？
>
> 面试官让你设计一个 key-value store，本质上就是设计一个**像 Redis 或 DynamoDB 这样的数据库**。
>
> 它支持两个操作：
>
> ```
> put(key, value)   // 存数据
> get(key)          // 取数据
> ```
>
> 听起来很简单对吧？但当数据量巨大、服务器有几千台、每天有几亿次请求时，问题就复杂了。
>
> ------
>
> ## 第一步：面试开始 — 澄清需求（必做！）
>
> **永远不要直接开始设计，先问需求。**
>
> 你要问的问题，以及本题的标准答案：
>
> | 你问的问题         | 本题要求                             |
> | ------------------ | ------------------------------------ |
> | 数据规模多大？     | 需要支持大数据，单机装不下           |
> | 对可用性要求高吗？ | 高可用，即使部分服务器挂了也要能响应 |
> | 对一致性要求高吗？ | 可调节（Tunable），不强求强一致      |
> | 延迟要求？         | 低延迟                               |
> | key-value 的大小？ | 小于 10 KB                           |
>
> 问完之后你说：
>
> > "好的，基于这些需求，我需要设计一个**分布式**的 key-value store，支持高可用、可扩展、低延迟，并且一致性可以根据场景调节。"
>
> ------
>
> ## 第二步：从单机方案开始，再引出问题
>
> ### 单机版（Single Server）
>
> 最简单的方案：**把所有 key-value 对存在一台服务器的内存哈希表里。**
>
> ```
> 内存哈希表：
> ┌─────────┬──────────────────┐
> │  key    │  value           │
> ├─────────┼──────────────────┤
> │ "user1" │ { name: "John" } │
> │ "user2" │ { name: "Amy" }  │
> │ "item1" │ { price: 99 }    │
> └─────────┴──────────────────┘
> ```
>
> 内存访问极快，但**问题是**：
>
> - 内存空间有限，数据量大了装不下
> - 服务器挂了，数据全丢了
> - 一台服务器扛不住海量请求
>
> > "所以单机方案不够，我们需要**分布式**方案，把数据分散到多台服务器上。"
>
> ------
>
> ## 第三步：分布式设计的核心挑战 — CAP 定理
>
> 在讲具体设计之前，必须先讲 CAP 定理。这是分布式系统的基础理论，面试里必须主动提。
>
> ### 什么是 CAP 定理？
>
> CAP 定理说：**分布式系统不可能同时满足以下三点，最多只能满足两点。**
>
> **C — Consistency（一致性）**
>
> > 所有客户端，无论连接到哪台服务器，看到的数据都是一样的（最新的）。
>
> **A — Availability（可用性）**
>
> > 任何时候发请求，都能收到响应，即使某些服务器挂了。
>
> **P — Partition Tolerance（分区容错性）**
>
> > 服务器之间的网络断了（网络分区），系统仍然能运行。
>
> ### 为什么 P 在现实中必须保留？
>
> 因为**网络故障是无法避免的**。服务器之间的网线可能断，机房可能断电，网络抖动随时发生。所以真实系统必须能容忍网络分区，P 不可放弃。
>
> 这意味着我们只能在 **C 和 A 之间做选择**：
>
> ```
> ┌─────────────────────────────────────────┐
> │              CAP 定理                    │
> │                                          │
> │         C（一致性）                       │
> │             △                            │
> │            / \                           │
> │           /   \                          │
> │          / CA  \    ← 现实中不存在         │
> │         /  区域  \                        │
> │        /─────────\                       │
> │       /     |     \                      │
> │      / CP   |  AP  \                     │
> │     /       |       \                    │
> │    A ───────┼─────── P                   │
> │（可用性）          （分区容错）              │
> │                                          │
> │  CP系统：保一致性，牺牲可用性 → 如 HBase    │
> │  AP系统：保可用性，牺牲一致性 → 如 Cassandra │
> └─────────────────────────────────────────┘
> ```
>
> ### 具体举个例子理解 CP vs AP
>
> 假设三台服务器 n1、n2、n3，n3 和其他两台的网络断了：
>
> ```
> n1 ←────→ n2
>  │    ✗    │     ← 这条线断了（网络分区）
>  └────✗────n3
> ```
>
> **如果选 CP（保一致性）：**
>
> - n1 和 n2 拒绝接受新的写请求（因为写了也同步不到 n3，会不一致）
> - 系统部分不可用，但数据是一致的
> - 适合：**银行系统**，账户余额必须准确，宁可报错也不能显示错误余额
>
> **如果选 AP（保可用性）：**
>
> - n1 和 n2 继续接受读写请求
> - n3 恢复后再同步数据（这段时间 n3 的数据可能是旧的）
> - 适合：**社交媒体**，你发的帖子别人晚几秒看到没关系
>
> > "对于我们的 key-value store，根据需求（tunable consistency），我们采用 **AP 系统**，默认使用最终一致性，但允许调节。"
>
> ------
>
> ## 第四步：核心组件设计
>
> 现在进入具体设计。按顺序讲这几个组件：
>
> ------
>
> ### 组件1：数据分区（Data Partition）— 用一致性哈希
>
> **问题：** 数据怎么分散到多台服务器上？
>
> **答案：** 使用**一致性哈希**（上一章学过的！）
>
> ```
>         s0
>        /    \
>      s7      s1
>     /            \
>   s6              s2
>     \            /
>      s5      s3
>        \    /
>         s4
> 
> key0 → 顺时针找到 s1 → 存在 s1
> ```
>
> **优点：**
>
> - 增删服务器时，只有少量 key 需要迁移
> - 可以用虚拟节点实现负载均衡
> - 性能强的服务器可以分配更多虚拟节点
>
> ------
>
> ### 组件2：数据复制（Data Replication）— 存 N 份副本
>
> **问题：** 如果一台服务器挂了，数据不就丢了？
>
> **答案：** 每份数据存 **N 个副本**（N 通常 = 3）。
>
> **规则：** key 落在哈希环上后，**顺时针找 N 台不同的服务器**，都存一份。
>
> ```
> key0 落在环上 →
>   顺时针第1台：s1（主副本）
>   顺时针第2台：s2（副本）
>   顺时针第3台：s3（副本）
> ```
>
> **注意：** 用虚拟节点时，要确保选的是 N 台**不同的物理服务器**，不能 3 个副本都在同一台物理机的虚拟节点上。
>
> **还有：** 为了防止机房整体故障（断电、自然灾害），副本要放在**不同的数据中心**。
>
> ------
>
> ### 组件3：一致性（Consistency）— Quorum 共识
>
> 这是最复杂的部分，但理解了之后非常清晰。
>
> **背景：** 数据有 3 个副本分散在不同服务器，读写的时候怎么保证数据是最新的？
>
> **引入三个参数：**
>
> ```
> N = 副本总数（通常 = 3）
> W = 写操作需要 W 台服务器确认才算成功
> R = 读操作需要从 R 台服务器读取才算成功
> ```
>
> **什么是"确认"？**
>
> 写的时候，你把数据发给 3 台服务器，服务器写成功后发回 ACK（acknowledgement，就是"我写好了"的确认信号）。
>
> ```
> 客户端
>   │
>   ▼
> 协调者（Coordinator）
>   ├──── put(key, val) ──→ s0（发回 ACK ✓）
>   ├──── put(key, val) ──→ s1（发回 ACK ✓）
>   └──── put(key, val) ──→ s2（还没回应...）
> 
> 如果 W=2，收到 s0 和 s1 的 ACK 就算写成功，不用等 s2
> ```
>
> **不同的 W、R 配置带来不同的效果：**
>
> | 配置        | 效果                         | 适用场景                 |
> | ----------- | ---------------------------- | ------------------------ |
> | W=1, R=1    | 读写都很快，但一致性差       | 对速度要求高，容忍旧数据 |
> | W=N, R=1    | 写慢（要等所有服务器），读快 | 读多写少                 |
> | W=1, R=N    | 写快，读慢（要读所有服务器） | 写多读少                 |
> | **W+R > N** | **强一致性保证**             | 需要精确数据             |
>
> **为什么 W+R > N 能保证强一致性？**
>
> 以 N=3, W=2, R=2 为例：
>
> ```
> 3 台服务器：s0, s1, s2
> 
> 写操作：成功写入 s0 和 s1（W=2）
> 读操作：从 s1 和 s2 读（R=2）
> 
> s1 是重叠的！s1 一定有最新数据。
> → 读到的结果中一定包含最新数据。
> ```
>
> **W + R > N 数学上保证：写集合和读集合必然有交集，交集里有最新数据。**
>
> ------
>
> ### 组件4：不一致性解决 — 向量时钟（Vector Clock）
>
> **问题：** 即使有副本，网络延迟可能导致不同副本的数据不一样，怎么解决冲突？
>
> **先理解问题是怎么产生的：**
>
> ```
> 初始状态：n1 和 n2 都存着 name = "john"
> 
> server1 把 n1 改成 name = "johnSanFrancisco"
> server2 把 n2 改成 name = "johnNewYork"
> 
> 现在 n1 和 n2 数据冲突了！哪个是对的？
> ```
>
> **解决方案：向量时钟（Vector Clock）**
>
> 向量时钟本质上是给每个数据版本打标签，记录"谁在什么时候改的"。
>
> 格式：`D([服务器, 版本号], [服务器, 版本号], ...)`
>
> **演示整个流程：**
>
> ```
> 步骤1：客户端写入，Sx 处理
>   D1([Sx, 1])   ← Sx 第一次写
> 
> 步骤2：客户端更新，还是 Sx 处理
>   D2([Sx, 2])   ← Sx 第二次写（版本号+1）
> 
> 步骤3：客户端基于 D2 更新，Sy 处理
>   D3([Sx, 2], [Sy, 1])   ← Sy 第一次写
> 
> 步骤4：另一个客户端也基于 D2 更新，Sz 处理
>   D4([Sx, 2], [Sz, 1])   ← Sz 第一次写
> 
> ⚠️ 冲突！D3 和 D4 都是从 D2 派生出来的，但互相不知道对方。
> 
> 步骤5：客户端发现冲突，手动合并，Sx 处理
>   D5([Sx, 3], [Sy, 1], [Sz, 1])
> ```
>
> **如何判断有没有冲突？**
>
> 规则：如果版本 X 的每个服务器版本号都 ≤ 版本 Y 对应的版本号，则 X 是 Y 的祖先（无冲突）。
>
> ```
> D([s0,1], [s1,1]) 是 D([s0,1], [s1,2]) 的祖先
> 因为 s0: 1≤1 ✓，s1: 1≤2 ✓  → 无冲突
> 
> D([s0,1], [s1,2]) 和 D([s0,2], [s1,1]) 有冲突
> 因为 s0: 1<2，但 s1: 2>1  → 互相不是祖先 → 冲突！
> ```
>
> **向量时钟的缺点：**
>
> - 客户端需要实现冲突解决逻辑，复杂
> - 向量时钟记录可能越来越长（解决：设置最大长度，删除最旧的记录）
>
> ------
>
> ### 组件5：故障检测 — Gossip 协议
>
> **问题：** 怎么知道哪台服务器挂了？
>
> **错误方法：** 每台服务器互相检测（all-to-all），服务器多了通信量爆炸。
>
> **正确方法：Gossip 协议**（像八卦/谣言一样传播信息）
>
> **工作原理：**
>
> ```
> 每台服务器维护一个成员列表：
> ┌──────────┬────────────┬──────────────┐
> │ 服务器ID  │ 心跳计数器  │ 最后更新时间  │
> ├──────────┼────────────┼──────────────┤
> │   s0     │   1023     │  12:00:01    │
> │   s1     │   1022     │  12:00:10    │
> │   s2     │    299     │  11:58:02  ← 很久没更新了！│
> │   s3     │    231     │  12:00:12    │
> └──────────┴────────────┴──────────────┘
> ```
>
> - 每台服务器定期给自己的心跳计数器 +1
> - 定期把自己的成员列表发给几台**随机**服务器
> - 收到别人的列表后，更新自己的列表（取最新值）
> - 如果某台服务器的心跳计数器**很久没变**，标记它为疑似宕机
> - 多台服务器确认后，正式标记为宕机，信息扩散出去
>
> ```
> s0 发现 s2 心跳很久没更新
>   → s0 告诉 s1 和 s3："s2 可能挂了"
>   → s1 和 s3 检查自己的记录，确认
>   → 大家都标记 s2 为 offline
> ```
>
> ------
>
> ### 组件6：故障处理 — 临时 vs 永久
>
> #### 临时故障：Sloppy Quorum + Hinted Handoff
>
> **问题：** 严格 Quorum 的话，某台服务器挂了，读写就被阻塞了（因为凑不够 W 或 R）。
>
> **解决：Sloppy Quorum（宽松 Quorum）**
>
> 不强求必须是那 N 台固定的服务器，而是从环上找 **W 个健康的服务器**写入，**R 个健康的服务器**读取，跳过宕机的服务器。
>
> **配套：Hinted Handoff（暗示移交）**
>
> ```
> 正常状态：s0, s1, s2 存副本
> 
> s2 宕机了：
>   → s3 临时代替 s2 存数据
>   → s3 记录："这份数据是给 s2 的，等它回来再还给它"
> 
> s2 恢复了：
>   → s3 把这段时间的数据"移交"给 s2
>   → 数据最终一致
> ```
>
> #### 永久故障：Merkle Tree（反熵）
>
> **问题：** 如果服务器永久挂了，换了一台新的，怎么同步数据？不可能把所有数据都比较一遍。
>
> **解决：Merkle Tree（哈希树）**
>
> Merkle Tree 是一种**能快速找出哪些数据不一样**的树形结构。
>
> **构建过程：**
>
> ```
> 假设 key 空间是 1-12，分成 4 个桶：
> 
> 桶1: key 1-3   桶2: key 4-6   桶3: key 7-9   桶4: key 10-12
> 
> 步骤1：每个 key 计算哈希
>   桶1: key1→h1, key2→h2, key3→h3
> 
> 步骤2：每个桶内所有 key 的哈希合并，得到桶的哈希
>   桶1哈希 = hash(h1+h2+h3) = 6901
>   桶2哈希 = 6773
>   桶3哈希 = 8601   ← server1 是 8601，server2 是 7975（不一样！）
>   桶4哈希 = 7812
> 
> 步骤3：向上合并，得到根节点
>              根
>            /     \
>       左子树     右子树
>        / \         / \
>     6901 6773  [8601] 7812
>                 ↑
>            这里不一样
> ```
>
> **比较两台服务器：**
>
> ```
> server1 根哈希：5357
> server2 根哈希：9213
>   → 不一样！有数据差异
> 
> 向下找：
>   左子树哈希一样？ → 一样，左边没问题
>   右子树哈希一样？ → 不一样，右边有问题
>   继续向下 → 找到是桶3（key 7-9）不一样
> 
> 只需要同步桶3的数据，不用全量同步！
> ```
>
> **好处：** 同步的数据量只和差异成正比，而不是和总数据量成正比。
>

> ## 第六步：写路径和读路径
>
> ### 写路径（Write Path）
>
> ```
> 客户端发 put(key, val)
>           │
>           ▼
>    1. 先写 Commit Log（提交日志）到磁盘
>       → 目的：防止服务器崩溃时数据丢失
>           │
>           ▼
>    2. 写入内存缓存（Memory Cache）
>       → 内存访问快，立即可读
>           │
>           ▼
>    3. 内存满了 → Flush 到磁盘的 SSTable
>       （SSTable = Sorted String Table，有序的键值对文件）
> 
> ┌─────────────────────────────────────┐
> │  MEMORY                             │
> │  ┌──────────────┐ ┌──────────────┐  │
> │  │ Commit Log   │ │Memory Cache  │  │
> │  │（持久化日志）  │ │（快速读写）   │  │
> │  └──────────────┘ └──────┬───────┘  │
> └─────────────────────────┼───────────┘
>                   Flush   │
>                           ▼
> ┌─────────────────────────────────────┐
> │  DISK                               │
> │  ┌──────────────────────────────┐   │
> │  │  SSTable  SSTable  SSTable   │   │
> │  │（有序键值对文件，不可变）      │   │
> │  └──────────────────────────────┘   │
> └─────────────────────────────────────┘
> ```
>
> ### 读路径（Read Path）
>
> ```
> 客户端发 get(key)
>           │
>           ▼
>    1. 先查内存缓存
>       ├── 找到了 → 直接返回 ✓
>       └── 没找到 → 去磁盘找
>                 │
>                 ▼
>    2. 查 Bloom Filter
>       （Bloom Filter：一种数据结构，能快速判断"这个 key 可能在哪个 SSTable 里"）
>                 │
>                 ▼
>    3. 查对应的 SSTable
>                 │
>                 ▼
>    4. 返回数据给客户端
> ```
>
> **什么是 Bloom Filter？**
>
> Bloom Filter 就像一个"目录"，不用翻遍所有 SSTable，它能告诉你"这个 key **可能**在 SSTable-3 里"，你只需要去查 SSTable-3。
>
> > 注意：Bloom Filter 可能有误报（说"在"但其实不在），但**不会漏报**（说"不在"就一定不在）。
>

> ## 第四步：API 设计

> ### 什么是 API？

> | **GET**    | 读取已有东西 | 看一篇文章   |
> | **PUT**    | 更新已有东西 | 修改文章内容 |
> | **DELETE** | 删除东西     | 删掉文章     |

> ## 第五步：高层设计（High-Level Design）
>
> > **面试话术**："Let me walk through each functional requirement and build up the system step by step."
> >
> > **关键原则**：从最简单的架构开始，一步步演进。不要一上来就画出终极版。
>
> ### 5.1 创建短链的流程
>
> **最简架构三件套**：
>
> ```
> Client（用户浏览器/App）
>     ↓ POST /urls
> Primary Server（主服务器）
>     ↓ 存储映射
> Database（数据库）
> ```
>
> **流程**：
>
> 1. 用户在网页上输入长 URL，点击"缩短"
> 2. 客户端发送 `POST /urls` 请求到 Primary Server
> 3. Primary Server **验证** URL 格式是否合法
> 4. Primary Server **生成短码**（short code）—— 具体怎么生成先不管，后面深入讨论
> 5. 如果用户指定了自定义别名，先**检查**数据库里有没有重复，没有就用它
> 6. 把"短码 → 长 URL"的映射**写入数据库**
> 7. 返回短链接给用户
>
> > **关于去重**：同一个长 URL 是否允许生成多个短链？大多数 URL 缩短服务**允许**，因为不同用户可能需要不同的过期时间、独立的分析数据、不同的自定义别名。
>
> ### 5.2 跳转还原的流程
>
> > 记住，短链的域名是**我们自己的**！比如 `short.ly/abc123`，所有发往 `short.ly` 的请求都会打到我们的服务器。
>
> ```
> 用户浏览器 → GET /abc123 → Primary Server → 查数据库 → 返回 302 重定向
> ```
>
> **流程**：
>
> 1. 用户点击短链 `short.ly/abc123`
> 2. 浏览器发送 `GET /abc123` 到我们的服务器
> 3. Primary Server 拿到短码 `abc123`，去数据库查对应的长 URL
> 4. 如果找到了且没过期 → 返回 **HTTP 302 重定向**
> 5. 如果已过期 → 返回 **HTTP 410 Gone**（表示"这个资源已经不存在了"）
> 6. 浏览器收到 302 后，**自动跳转**到原始长 URL，用户无感知
>
> ### 301 vs 302 重定向 — 必须掌握
>
> > **什么是 HTTP 重定向？** 服务器告诉浏览器："你要的东西不在这里，去那个地方找。"浏览器会**自动**跳过去，用户完全无感。
>
> | 状态码  | 名称              | 含义       | 浏览器行为                                         |
> | ------- | ----------------- | ---------- | -------------------------------------------------- |
> | **301** | Moved Permanently | 永久搬家了 | **缓存**这个重定向，下次直接去新地址，不再问服务器 |
> | **302** | Found             | 临时在别处 | **不缓存**，每次都先来问服务器                     |
>
> **我们选 302，三个理由**：
>
> 1. **保持控制权**：如果要修改或删除短链，302 保证每次请求都经过我们的服务器，我们随时可以改
> 2. **避免缓存问题**：301 被浏览器缓存后，即使我们删了短链，用户浏览器仍然会自动跳转到旧地址
> 3. **支持未来扩展**：如果以后要做点击统计，302 让每次点击都经过我们的服务器，方便记录
>
> > **面试金句**："I'd use 302 Found because it gives us more control — we can update, expire, or delete links at any time, and it keeps the door open for future analytics."
>
> ### 过期 URL 的清理
>
> - **查询时检查**：每次跳转查询时，比对当前时间和过期时间，过期就返回 410
> - **后台定期清理**：跑一个 background job，定期删除过期的行（可以做也可以不做，过期数据留着也没大问题）
> - **缓存 TTL**：缓存的过期时间要 ≤ URL 的过期时间，防止缓存里存着已过期的链接还在跳转
>

>
> > **面试话术**："Now that we have a basic working system, let me dive deeper into the key challenges based on our non-functional requirements."
>
> ### Deep Dive 1：如何保证短码唯一？
>
> > **面试话术**："In our high-level design, I abstracted away short code generation. Let me now explore different approaches and their tradeoffs."
>
> #### 方案 A：取前缀（❌ 反面教材，用来展示思考过程）
>
> 直接取长 URL 的前 N 个字符作为短码。
>
> 比如 `www.linkedin.com/in/evan-king` → 短码 `www.link`
>
> **问题**：任何以 `www.link` 开头的 URL 都会冲突。完全不可行。
>
> > **面试中为什么要提这个？** 展示你的思考过程是"从简单到复杂"层层递进的，而不是直接蹦出最优解。
>
> #### 方案 B：哈希函数（⚠️ 可行但有缺陷）
>
> ##### 什么是哈希函数？
>
> 哈希函数就像一台"指纹机"：你给它任何输入，它都会输出一个**固定长度的字符串**（像指纹一样）。
>
> 核心特性：
>
> - **确定性**：同样的输入，永远得到同样的输出
> - **高熵（随机性）**：不同输入的输出看起来完全随机，极少碰撞
> - **不可逆**：从输出无法推算出输入
>
> 比如 SHA-256 就是一种常用的哈希函数。
>
> ##### 什么是 Base62 编码？
>
> Base62 用 62 个字符（a-z、A-Z、0-9）来表示数字。
>
> **为什么不用 Base64？** Base64 多了 `+` 和 `/` 两个字符。`/` 在 URL 里是路径分隔符，`+` 在 URL 查询参数里会被解读为空格，所以都不能用。
>
> ##### 方案 B 的流程
>
> ```
> 长 URL → SHA-256 哈希 → Base62 编码 → 取前 8 个字符作为短码
> ```
>
> 为什么取 8 个字符？62^8 ≈ 218 万亿种可能，碰撞概率极低。
>
> ##### 碰撞处理
>
> 虽然概率低，但不是零。处理方案：
>
> 1. 数据库的短码列加 **UNIQUE 约束**（数据库层面保证不会有两行有相同短码）
> 2. 插入时如果报唯一冲突错误，**加随机盐（salt）重新哈希**，最多重试 3-5 次
> 3. 超过重试次数就报错
>
> **缺点**：碰撞检测需要额外的数据库查询，增加延迟和复杂度。而且要短码足够短就得接受更高的碰撞率，这是个矛盾。
>
> #### 方案 C：自增计数器 + Base62（✅ 推荐方案）
>
> ##### 核心思想
>
> 每来一个新 URL，计数器 +1，然后把计数器的值用 Base62 编码成短码。
>
> ```
> 第 1 个 URL → 计数器=1 → Base62="1" → 短码 "1"
> 第 1000 个 URL → 计数器=1000 → Base62="g8" → 短码 "g8"
> 第 10 亿个 URL → 计数器=1000000000 → Base62="15ftgG" → 短码 "15ftgG"（才 6 位！）
> ```
>
> ##### 为什么用 Redis 管理计数器？
>
> > **什么是 Redis？** Redis 是一个内存数据库（所有数据存在内存里，速度极快）。它的一个关键特性是**单线程**——一次只处理一个命令，永远不会两个命令同时执行。
>
> Redis 的 `INCR` 命令：**原子性地**把计数器 +1 并返回新值。
>
> > **什么是原子性？** 就是"要么完整执行，要么完全不执行"，中间不会被打断。好比你从 ATM 取钱，要么钱出来且余额减少，要么什么都没发生。不可能出现"钱出来了但余额没变"的情况。
>
> 因为 Redis 是单线程的，即使两个服务器**同时**调用 `INCR`，一个拿到 1000，另一个必然拿到 1001。**绝对不会重复**。
>
> ##### 方案 C 的优势
>
> | 优点         | 说明                                   |
> | ------------ | -------------------------------------- |
> | **零碰撞**   | 每个计数器值天然唯一，不需要检查数据库 |
> | **高效**     | 计数器 +1 再编码，计算量极小           |
> | **短码够短** | 10 亿个 URL 只需 6 个字符              |
>
> ##### 方案 C 的短码长度估算
>
> | URL 数量 | Base62 编码长度         |
> | -------- | ----------------------- |
> | 100 万   | 4 个字符                |
> | 10 亿    | 6 个字符                |
> | 560 亿   | 6 个字符（62^6 的上限） |
> | 3.5 万亿 | 7 个字符（62^7 的上限） |
>
> > **面试金句**："Even at 1 billion URLs, base62 encoding only produces a 6-character code. We have plenty of room to scale."
>
> ##### 方案 C 的潜在问题
>
> **问题 1：短码可预测**
>
> 计数器是递增的，攻击者可以枚举 `short.ly/1`、`short.ly/2`、`short.ly/3`... 来发现所有 URL。
>
> **解决**：在 Base62 编码前，用一个**可逆变换**（如 XOR 加密密钥）打乱顺序。或者接受这个 tradeoff，因为短链本来就是用来分享的，不需要保密。
>
> **问题 2：分布式环境下的计数器同步**
>
> 多台服务器需要共享同一个计数器。这就是为什么用 Redis（集中式、单线程、原子操作）。后面在"扩展性"部分会详细讨论。
>
> ##### 自定义别名和生成短码的冲突防范
>
> 用户自定义的别名（如 `my-brand`）可能和未来计数器生成的短码碰撞。
>
> **解决方案**：
>
> - 给生成的短码加一个固定前缀（比如 `_`），自定义别名不允许以 `_` 开头
> - 或者把它们存到不同的命名空间（namespace）
>
> ------
>
> ### Deep Dive 2：如何保证跳转速度快？
>
> > **面试话术**："Our non-functional requirement is sub-100ms redirects. Let me walk through how we can achieve that."
>
> #### 问题分析
>
> 没有优化的话，每次跳转都要去数据库查，而数据库查询意味着**磁盘 I/O**。
>
> ##### 什么是全表扫描？
>
> 如果数据库不知道数据在哪，它只能**一行一行扫描**整个表来找匹配的短码。10 亿行数据这么找，慢到不能用。就像在一本没有目录的百科全书里找一个词。
>
> #### 优化 1：数据库索引（Database Index）
>
> ##### 什么是索引？
>
> 索引就是数据库的"目录"。
>
> 想象一本 1000 页的书。没有目录，找某个章节要从头翻到尾。有了目录，直接翻到目录页，找到"第 5 章 → 第 342 页"，一下就到了。
>
> 数据库索引的原理一样：它在短码列上建一个**排好序的快速查找结构**（通常是 B-tree），让查找时间从 O(n) 降到 O(log n)。
>
> > **什么是 O(log n)？** 假设有 10 亿条数据，O(n) 要查 10 亿次，O(log n) 只要查大约 30 次。因为每次查找都能排除一半的数据（类似二分查找）。
>
> ##### 实际做法
>
> 把短码设为表的 **Primary Key（主键）**。这样做一举两得：
>
> - 自动创建索引，查询飞快
> - 自动保证唯一性，数据库不允许插入重复的短码
>
> #### 优化 2：内存缓存（In-Memory Cache）— 核心优化
>
> ##### 为什么光有索引不够？
>
> 来算一笔账：
>
> ```
> 1 亿日活 × 每人每天 5 次跳转 = 5 亿次跳转/天
> 5 亿 ÷ 86400 秒 ≈ 5,787 次/秒（平均）
> 高峰期按 100 倍算 ≈ 60 万次/秒
> ```
>
> 即使有索引，**单台数据库处理 60 万次/秒的读请求也很吃力**。
>
> ##### 什么是缓存？
>
> 缓存就是"把经常要用的东西放在手边"。
>
> 类比：你每天早上都查同一个单词，与其每次翻字典，不如把它写在便利贴上贴在桌上。下次直接看便利贴就行。
>
> 在系统设计里，缓存就是一个**内存数据库**（比如 Redis 或 Memcached），存放最常访问的"短码 → 长 URL"映射。
>
> ##### 不同存储介质的速度对比
>
> | 存储介质            | 单次访问时间 | 每秒操作数  |
> | ------------------- | ------------ | ----------- |
> | **内存（RAM）**     | ~0.0001 ms   | 数百万次    |
> | **SSD**             | ~0.1 ms      | ~10 万次    |
> | **HDD（机械硬盘）** | ~10 ms       | ~100-200 次 |
>
> 内存比 SSD 快 **1000 倍**，比 HDD 快 **10 万倍**。
>
> ##### 缓存的工作流程
>
> ```
> 用户请求 GET /abc123
>     ↓
> Server 先查缓存
>     ├─ 缓存命中（Cache Hit）→ 直接返回长 URL，超快！
>     └─ 缓存未命中（Cache Miss）→ 去数据库查
>         → 查到后存入缓存（下次就命中了）
>         → 返回长 URL
> ```
>
> ##### 缓存的挑战
>
> | 挑战                               | 说明                                                         |
> | ---------------------------------- | ------------------------------------------------------------ |
> | **缓存失效（Cache Invalidation）** | URL 被删除或更新后，缓存里的旧数据怎么办？好在 URL 短链几乎不会被修改，这个问题不严重 |
> | **冷启动（Cold Start）**           | 缓存刚启动时是空的，所有请求都要打到数据库，直到缓存"热起来" |
> | **内存有限**                       | 不可能把 10 亿条都缓存，需要选择淘汰策略                     |
> | **淘汰策略**                       | 推荐 **LRU（Least Recently Used）**——最近最少使用的先踢出去  |
>
> > **什么是 LRU？** 想象你的书桌只能放 10 本书。新拿来一本书时，桌子满了就把最久没碰过的那本放回书架。这就是 LRU。
>
> > **面试金句**："Given the heavy read pattern, an in-memory cache like Redis with LRU eviction can absorb the vast majority of redirect traffic, keeping our p99 latency well under 100ms."
>
> #### 优化 3：CDN 和边缘计算（可选加分项）
>
> ##### 什么是 CDN？
>
> CDN（Content Delivery Network）是分布在全球各地的服务器网络。用户访问时，由离他最近的服务器响应。
>
> 类比：亚马逊的仓库遍布全国，你下单后从最近的仓库发货，而不是所有包裹都从一个中心仓库发。
>
> ##### 应用到 URL 短链
>
> 把短码→长 URL 的映射**缓存到 CDN 节点**上。用户点击短链时，CDN 直接返回重定向，**请求根本不需要打到我们的服务器**。
>
> 可以用 **Cloudflare Workers** 或 **AWS Lambda@Edge** 把跳转逻辑部署到边缘。
>
> ##### CDN 的 Tradeoff
>
> | 优点                   | 缺点                       |
> | ---------------------- | -------------------------- |
> | 延迟极低（物理距离近） | 跨所有节点的缓存失效很复杂 |
> | 大幅减轻源服务器压力   | 配置和调试更复杂           |
> |                        | 成本更高（流量越大越贵）   |
>
> > **面试金句**："CDN adds cost and complexity, but for a URL shortener with extremely high read traffic, the latency benefit can be significant. Whether it's worth it depends on the company's scale and cost sensitivity."
>
> ------
>
> ### Deep Dive 3：如何扩展到 10 亿 URL + 1 亿 DAU？
>
> > **面试话术**："Let me address scalability. We've already introduced caching for reads, now let's think about the overall system."
>
> #### 数据库容量估算
>
> 先算一下数据量，看看需不需要分库：
>
> | 字段         | 大小                                          |
> | ------------ | --------------------------------------------- |
> | 短码         | ~8 bytes                                      |
> | 长 URL       | ~100 bytes                                    |
> | 创建时间     | ~8 bytes                                      |
> | 自定义别名   | ~100 bytes                                    |
> | 过期时间     | ~8 bytes                                      |
> | **每行合计** | ~200 bytes，考虑其他元数据按 **500 bytes** 算 |
>
> ```
> 10 亿行 × 500 bytes = 500 GB
> ```
>
> 500 GB 对现代 SSD 来说**完全没问题**。一台 PostgreSQL 实例就能撑住。
>
> > **面试金句**："500GB for 1 billion URLs is well within the capacity of a single Postgres instance. We don't need to shard the database at this scale."
>
> #### 数据库选型
>
> 写入量估算：每天 10 万条新 URL ÷ 86400 秒 ≈ **1 次写入/秒**。
>
> 这个写入量**任何数据库都能轻松应对**。PostgreSQL、MySQL、DynamoDB 都行。面试里选你最熟悉的，没经验就选 **PostgreSQL**。
>
> #### 数据库高可用
>
> > **面试话术**："What if the database goes down?"
>
> 两个策略：
>
> 1. **数据库复制（Replication）**：创建数据库的多个副本分布在不同服务器上。主库挂了，切换到副本继续服务。
> 2. **定期备份（Backup）**：周期性快照备份到独立存储位置，用于灾难恢复。
>
> #### 服务拆分：读写分离
>
> 回到核心洞察：读 >> 写。所以我们把 Primary Server **拆成两个独立服务**：
>
> ```
>              ┌─────────────────┐
>              │   Read Service   │ ← 处理跳转（GET /{short_code}）
>              │  （可独立扩展）   │
>              └─────────────────┘
> Client ──→ 
>              ┌─────────────────┐
>              │  Write Service   │ ← 处理创建短链（POST /urls）
>              │  （可独立扩展）   │
>              └─────────────────┘
> ```
>
> ##### 什么是微服务架构？
>
> 把一个大服务拆成多个小服务，每个只负责一件事。好处是可以**独立扩展**——读流量大就多加几台 Read Service，写流量不大就保持少量 Write Service。
>
> ##### 什么是水平扩展？
>
> **垂直扩展（Scale Up）**：给一台机器加 CPU、内存（有上限，且越来越贵）
>
> **水平扩展（Scale Out）**：加更多机器来分担负载（理论上无上限）
>
> 类比：垂直扩展是让一个服务员跑得更快，水平扩展是多雇几个服务员。
>
> #### 分布式环境下的计数器问题
>
> 水平扩展 Write Service 后，多台机器都要生成短码。计数器必须是**全局唯一的单一来源**。
>
> **解决方案**：用一个**集中式 Redis 实例**存储计数器。所有 Write Service 实例都向这个 Redis 请求下一个计数器值。
>
> ```
> Write Service A ──→ Redis INCR → 得到 1000
> Write Service B ──→ Redis INCR → 得到 1001（保证不重复）
> ```
>
> #### 计数器批量分配（Counter Batching）— 性能优化
>
> 每次创建短链都要请求一次 Redis，虽然 Redis 很快，但能不能更高效？
>
> **批量分配**：每台 Write Service 一次从 Redis 领取 1000 个计数器值，本地慢慢用，用完了再去领。
>
> ```
> Write Service A → 向 Redis 请求 → Redis 返回 1000-1999（Redis 内部计数器 +1000）
> Write Service A 在本地依次使用 1000, 1001, 1002...
> 用完 1999 后再去领 2000-2999
> ```
>
> **好处**：
>
> - 大幅减少 Redis 的网络请求（从每次创建 1 次请求降到每 1000 次创建 1 次请求）
> - Redis 压力更小，整体性能更高
>
> **代价**：
>
> - 如果某台 Write Service 挂了，它手里还没用完的计数器值就"浪费"了（但这只是浪费几个数字，不影响正确性）
>
> #### Redis 高可用
>
> - 使用 **Redis Sentinel** 或 **Redis Cluster**，实现自动故障切换（failover）
> - 单台 Redis 能处理 10 万+次操作/秒，配合批量分配后远远超过需求
> - 如果 Redis 在复制之前宕机，可能丢失最近的几个计数器值。但因为我们只需要**唯一性**而不需要**连续性**，丢几个数字完全可以接受
> - 数据库的 UNIQUE 约束是**最后的安全网**，即使计数器出问题，数据库也会拒绝重复插入
>
> #### 多区域部署（Multi-Region）
>
> 如果要在全球多个区域部署：
>
> - 给每个区域分配**不重叠的计数器范围**（比如区域 A 用 0-10 亿，区域 B 用 10 亿-20 亿）
> - 写入走本地区域的 Redis，不需要跨区域协调
> - 读取通过分布式缓存全球服务
>
> ------
>
> ## 最终架构总览
>
> ```
>                           ┌──────────┐
>                           │   CDN    │ ← 热门短链直接在边缘跳转（可选）
>                           └────┬─────┘
>                                │
>                           ┌────▼─────┐
>                   ┌───────│  Client  │───────┐
>                   │       └──────────┘       │
>                   │                          │
>            GET /{code}                POST /urls
>                   │                          │
>          ┌────────▼────────┐     ┌───────────▼──────────┐
>          │  Read Service   │     │   Write Service       │
>          │  (多实例水平扩展) │     │  (多实例水平扩展)      │
>          └────────┬────────┘     └──────┬──────────┬─────┘
>                   │                     │          │
>          ┌────────▼────────┐            │   ┌──────▼──────┐
>          │  Redis Cache    │            │   │   Redis     │
>          │  (缓存热门映射)  │            │   │  (全局计数器) │
>          └────────┬────────┘            │   └─────────────┘
>                   │ Cache Miss          │
>                   │                     │
>          ┌────────▼─────────────────────▼──┐
>          │         PostgreSQL               │
>          │  (存储短码→长URL的映射)           │
>          │  主键=短码, 自动索引+唯一约束     │
>          │  Replication for HA             │
>          └─────────────────────────────────┘
> ```

> 1. **读写比**："This system has an extremely high read-to-write ratio, roughly 1000:1, so our design should heavily optimize for reads."
> 2. **选 302**："I'd use 302 Found to maintain control over redirects — it lets us update, expire, or delete links, and keeps the door open for analytics."
> 3. **计数器方案**："A monotonically increasing counter with base62 encoding guarantees uniqueness with zero collision overhead. Redis INCR is ideal because it's single-threaded and atomic."
> 4. **容量估算**："At 500 bytes per row and 1 billion URLs, that's 500GB — well within a single Postgres instance."

> 8. **CDN tradeoff**："CDN adds cost and complexity, but for high read traffic, the latency improvement is significant. It's a tradeoff between cost and performance."

> ## 🧠 这道题的核心难点是什么？
>
> 在开始之前，先理解这道题的灵魂：
>
> > **Fan-out 问题**：一个人发了一条帖子，他的100万粉丝每人的首页都要更新。这个"一变百万"的扩散过程叫 **Fan-out（扇出）**。
>
> 整道题围绕这一个问题展开：**何时做这个扩散？写的时候做？还是读的时候做？**
>
> ------
>
> ## 第一步：澄清需求 ⏱️ 约5分钟
>
> 开场说：
>
> > *"在开始设计之前，我想先确认核心需求。"*
>
> ------
>
> ### ✅ 功能需求
>
> | 功能     | 说明                                                   |
> | -------- | ------------------------------------------------------ |
> | 发帖     | 用户可以创建新帖子                                     |
> | 关注     | 用户可以关注其他用户（单向，不是双向好友）             |
> | 查看Feed | 看到自己关注的人发的帖子，按时间倒序（最新的在最上面） |
> | 翻页     | 可以往下滑，看更早的帖子（无限滚动）                   |
>
> **不做的功能**：点赞、评论、帖子隐私设置
>
> ------
>
> ### ✅ 非功能需求
>
> | 指标       | 要求                                 | 解释                                            |
> | ---------- | ------------------------------------ | ----------------------------------------------- |
> | 可用性     | 高可用，优先可用性而非一致性         | 宁愿给用户看旧数据，也不要让系统挂掉            |
> | 延迟       | 发帖和查看Feed都要 < 500ms           | 半秒内响应                                      |
> | 规模       | 20亿用户                             |                                                 |
> | 关注数量   | 可以关注任意多人，可以被任意多人关注 | 这句话是关键！意味着有人有几千万粉丝            |
> | 一致性容忍 | 允许最多1分钟的延迟                  | 刚发的帖子，粉丝Feed最多1分钟后才看到，可以接受 |
>
> **⚠️ 主动说出这句话：**
>
> > *"这里有一个关键观察：这个系统**读远多于写**。用户每天刷几十次Feed，但发帖频率很低。所以读性能优化是重点。另外，'可以被任意多人关注'意味着我们需要特别处理大V账号的扇出问题，这将是这道题最复杂的部分。"*
>
> ------
>
> ## 第二步：定义核心实体 ⏱️ 约2分钟
>
> ```
> 1. User（用户）        — 系统里的人
> 2. Follow（关注关系）  — 谁关注了谁（单向）
> 3. Post（帖子）        — 用户发的内容
> ```
>
> > 💡 **为什么要单独列Follow实体？** 因为"关注关系"是独立的数据，不属于用户表也不属于帖子表。比如"张三关注了李四"这条关系需要单独存储。
>
> ------
>
> ## 第三步：定义API ⏱️ 约5分钟
>
> ------
>
> ### API 1：发帖
>
> ```
> POST /posts
> {
>   "content": { }    ← 内容留空，支持文字/图片等富媒体
> }
> 
> 返回：
> {
>   "postId": "post_12345"
> }
> ```
>
> ------
>
> ### API 2：关注用户
>
> ```
> PUT /users/{userId}/follow
> 
> 返回：200 OK
> ```
>
> > 💡 **为什么用PUT而不是POST？**
> >
> > - `POST` 每次都创建新资源，重复点关注会出问题
> > - `PUT` 是**幂等的**（idempotent）：执行多次和执行一次效果一样。用户手抖点了两次关注按钮，也没问题
> >
> > 💡 **什么是幂等？** 比如"把灯打开"是幂等操作——灯已经开着，你再按一次开关，结果还是灯开着，没有副作用。
>
> ------
>
> ### API 3：获取Feed（核心）
>
> ```
> GET /feed?pageSize=20&cursor=2024-01-15T10:30:00Z
> 
> 返回：
> {
>   "items": [ Post, Post, Post... ],
>   "nextCursor": "2024-01-14T08:20:00Z"
> }
> ```
>
> > 💡 **什么是cursor（游标）分页？**
> >
> > 想象你在读一本书，你的书签（cursor）夹在第50页。下次打开书，直接从第51页开始读，不用从第1页重新翻。
> >
> > 这里的cursor是一个**时间戳**，代表"用户上次看到的最旧那条帖子的时间"。下次请求，只返回比这个时间更早的帖子。
> >
> > 为什么不用"第几页"（page=2）？因为Feed是动态的，如果你看第2页的时候刚好有人发了新帖，"第2页"的内容就变了，会出现重复或跳过。游标没有这个问题。
>
> ------
>
> ## 第四步：高层设计 ⏱️ 约15分钟
>
> > 原则：**先做一个能跑的最简单方案，承认它的问题，然后说"等下Deep Dive来解决"。**
>
> ------
>
> ### 功能1：发帖
>
> **架构：**
>
> ```
> 用户 → API网关/负载均衡器 → Post Service（多实例）→ DynamoDB（Posts表）
> ```
>
> > 💡 **什么是API网关/负载均衡器？**
> >
> > - **负载均衡器**：你开了3家一模一样的餐厅，门口有个服务员，来一个客人就指向最空闲的那家。这样没有哪家会被挤爆。
> > - **API网关**：更聪明的版本，还能做身份验证、限流、路由等。
>
> > 💡 **为什么用DynamoDB？** 它是AWS的NoSQL键值数据库，设计上可以无限水平扩展，适合海量读写。只要数据均匀分布在各个分区上，就能支撑极高吞吐量。
>
> **Posts表结构：**
>
> ```
> Posts表
> ┌──────────────┬──────────────┬─────────────────────┬─────────┐
> │   postId     │  creatorId   │      createdAt       │ content │
> ├──────────────┼──────────────┼─────────────────────┼─────────┤
> │  post_001    │  user_123    │  2024-01-15 10:30   │  "..."  │
> └──────────────┴──────────────┴─────────────────────┴─────────┘
> ```
>
> ------
>
> ### 功能2：关注用户
>
> **Follow表结构（存在DynamoDB）：**
>
> ```
> Follow表
> ┌──────────────────┬──────────────────┐
> │  userFollowing   │  userFollowed    │   ← 主键（复合主键）
> │  (分区键)         │  (排序键)         │
> ├──────────────────┼──────────────────┤
> │    user_A        │    user_B        │   ← A关注了B
> │    user_A        │    user_C        │   ← A关注了C
> │    user_B        │    user_A        │   ← B关注了A
> └──────────────────┴──────────────────┘
> ```
>
> > 💡 **什么是分区键（Partition Key）和排序键（Sort Key）？**
> >
> > 想象一个文件柜：
> >
> > - 分区键 = 抽屉标签（比如"A-关注的人"）。决定数据放在哪个抽屉
> > - 排序键 = 抽屉里文件的排列顺序（按字母/时间等）
> >
> > 分区键相同的数据放在同一个抽屉里，可以快速范围查询。
>
> **还需要一个反向索引（GSI）：**
>
> > 💡 **什么是GSI（Global Secondary Index，全局二级索引）？** 相当于给文件柜建了另一套目录系统。原始目录按"谁关注了别人"来找，GSI按"谁被别人关注"来找。同一份数据，两种查找方式。
>
> ```
> GSI（反向索引）：以userFollowed为分区键
> → 可以查询"user_B被哪些人关注了"
> ```
>
> **三种查询都能支持：**
>
> - 查"A是否关注了B" → 用分区键+排序键，O(1)
> - 查"A关注了哪些人" → 用分区键，范围查询
> - 查"B有哪些粉丝" → 用GSI的分区键，范围查询
>
> ------
>
> ### 功能3 & 4：查看Feed + 翻页
>
> **给Posts表加一个GSI：**
>
> ```
> Posts表的GSI：以creatorId为分区键，createdAt为排序键
> → 可以快速查"某用户发的所有帖子，按时间排序"
> ```
>
> **最简单（最朴素）的Feed生成流程：**
>
> ```
> 用户请求 GET /feed
>     ↓
> Feed Service
>     ↓
> Step 1：查Follow表 → 得到用户关注的所有人 [userB, userC, userD...]
>     ↓
> Step 2：对每个关注的人，查Posts表的GSI → 得到他们的帖子
>     ↓
> Step 3：把所有帖子合并，按时间排序，取最新的N条
>     ↓
> 返回给用户
> ```
>
> **主动说出问题（重要！）：**
>
> > *"这个方案能跑，但有明显的扩展性问题：如果用户关注了1000个人，Step 2就需要发1000个数据库请求；如果某个关注的人发了几万条帖子，数据量也很大。这是'读时扇出'（Fan-out on Read）问题。我在Deep Dive里来解决它，现在先继续。"*
>
> > 💡 **什么是读时扇出（Fan-out on Read）？** 你去餐厅点菜，服务员说"等一下，我要去菜市场买菜、去水产市场买鱼、去农场拿鸡蛋……"每次点菜都实时去凑食材，太慢了。这就是读时扇出——读的时候触发大量工作。
>
> ------
>
> ## 第五步：Deep Dives ⏱️ 约20分钟
>
> ------
>
> ### Deep Dive 1：如何处理关注了很多人的用户？（Fan-out on Read问题）
>
> **解决思路：变"读时计算"为"写时预计算"**
>
> > 💡 **核心转变：Fan-out on Write（写时扇出）**
> >
> > 不要等用户来请求Feed时再拼装，而是**每次有人发帖，就立刻把这条帖子推送到所有粉丝的"预计算Feed"里**。用户来请求时，直接读预计算好的结果，极快！
> >
> > 就像餐厅提前备菜，顾客来了直接上菜，不用现做。
>
> **新增：PrecomputedFeed表**
>
> ```
> PrecomputedFeed表
> ┌──────────────┬──────────────────────────────────────────┐
> │   userId     │              postIds（列表）               │
> ├──────────────┼──────────────────────────────────────────┤
> │  user_A      │  [post_999, post_998, post_997, ...]     │
> │              │  （最新200条，按时间倒序）                   │
> └──────────────┴──────────────────────────────────────────┘
> ```
>
> > 每个用户只存最近200条帖子ID（不存全部内容，节省空间）。用户翻页超过200条？再回退到查原始表。
>
> **存储估算（要在面试中说出来）：**
>
> ```
> 每个postId约10字节
> 每人存200条 → 2KB
> 20亿用户 → 2KB × 20亿 = 4TB
> ```
>
> > *4TB对于现代系统完全合理，成本极低。*
>
> **新的Feed读取流程（超简单）：**
>
> ```
> 用户请求 GET /feed
>     ↓
> 直接查PrecomputedFeed表，按userId取出帖子ID列表
>     ↓
> 批量查Posts表，取出帖子详情
>     ↓
> 返回！
> ```
>
> ------
>
> ### Deep Dive 2：如何处理拥有大量粉丝的大V？（Fan-out on Write的新问题）
>
> **新问题的产生：**
>
> > 大V发一条帖子 → 需要更新1000万个粉丝的PrecomputedFeed → **1000万次写操作，还要在1分钟内完成！**
>
> ------
>
> #### ❌ 方案A：发帖时直接写1000万条（同步写）
>
> > 就像发1000万封信，你一个人手写，根本写不完。
>
> **问题**：
>
> - 一台服务器无法建立1000万个数据库连接
> - 延迟无法接受
> - 某台服务器在处理大V的帖子时，其他普通用户的帖子没法处理（负载极度不均衡）
>
> ------
>
> #### ✅ 方案B：消息队列 + 异步Worker（**好方案**）
>
> > 💡 **什么是消息队列（Message Queue）？**
> >
> > 想象一个快递公司的分拣中心。有人寄了1000个包裹，不是让寄件人自己送，而是扔进队列，由一批快递员（Worker）慢慢处理。寄件人发完帖子就可以走了（异步），快递员在后台慢慢处理（不影响寄件人）。
> >
> > 这里用**Amazon SQS**（Simple Queue Service）。
>
> **新流程：**
>
> ```
> 大V发帖
>     ↓
> Post Service把任务放进SQS队列
> （"userId=大V, postId=xxx"）
>     ↓
> 大V立刻收到"发帖成功"响应（不等1000万条写完）
>     ↓
> 后台：一批Feed Worker从队列里取任务
>     ↓
> 每个Worker：查Follow表 → 找出这批粉丝 → 更新他们的PrecomputedFeed
>     ↓
> （在1分钟内完成，满足一致性要求）
> ```
>
> > 💡 **什么是at-least-once delivery（至少一次投递）？** 消息队列保证每条消息至少被处理一次，即使Worker中途挂了，消息也不会丢失（会被重新投递）。这比"只处理一次"更安全。
>
> **还有一个细节问题：**
>
> > 大V（1000万粉丝）和普通用户（100粉丝）的任务被放在同一个队列里，处理量差距太大。可以把大V的任务拆分成多个子任务（每个子任务处理1万粉丝），多个Worker并行处理。
>
> ------
>
> #### ✅✅ 方案C：混合策略（Hybrid，**最佳方案**）
>
> > 核心思路：**对普通用户用写时扇出，对大V用读时扇出，两者在读取时合并。**
>
> **具体做法：**
>
> 1. 在Follow表里给每条关注关系加一个标记：`isPrecomputed = false`（表示这个关注关系不做预计算）
> 2. 当粉丝数超过某个阈值（比如100万），把这个用户的所有关注关系标记为 `isPrecomputed = false`
> 3. 后台Worker在处理大V帖子时，跳过这些标记的关注关系，不更新对应粉丝的Feed
> 4. 用户请求Feed时：
>    - 从PrecomputedFeed表取出预计算好的帖子（来自普通用户）
>    - **同时**实时查询大V的最新帖子（Fan-out on Read，但只对少数大V做）
>    - 把两部分合并，排序后返回
>
> ```
> 用户请求 GET /feed
>     ↓
> Feed Service
>     ↓
> 读PrecomputedFeed（普通用户的帖子，已预计算）
>                     +
> 实时查大V的最新帖子（只查几个大V，可接受）
>     ↓
> 合并 + 排序
>     ↓
> 返回
> ```
>
> **为什么这是最优解：**
>
> | 情况                    | 策略               | 原因                          |
> | ----------------------- | ------------------ | ----------------------------- |
> | 普通用户发帖（100粉丝） | 写时扇出           | 只需写100条，很快             |
> | 大V发帖（1000万粉丝）   | 不预计算，读时合并 | 避免1000万次写操作            |
> | 用户读Feed              | 合并两部分         | 大V数量少，实时查询开销可接受 |
>
> ------
>
> ### Deep Dive 3：如何处理热门帖子的大量读请求？
>
> **问题：**
>
> > 一条帖子突然爆红，每秒被读取100万次。DynamoDB按分区存储数据，这条帖子的数据只在一个分区上，这个分区被打爆了（**热键问题，Hot Key**）。
>
> > 💡 **什么是热键（Hot Key）？** 数据库把数据分散存在不同机器上，但某一条数据（比如爆款帖子）被疯狂访问，对应的那台机器承受了所有压力，其他机器闲着。就像超市里只有一个收银台排了长队，其他都没人。
>
> ------
>
> #### ✅ 方案A：分布式缓存（**好方案**）
>
> > 💡 **什么是缓存（Cache）？** 把数据库的数据复制一份放在内存里（比如Redis），下次查同一个数据，直接从内存拿，不用去数据库。内存比数据库快1000倍。
>
> > 💡 **什么是TTL（Time To Live，生存时间）？** 缓存里的数据不能永远放着，要设置一个"保质期"。过期了自动删除，下次请求时重新从数据库获取最新数据。
>
> > 💡 **什么是LRU（Least Recently Used）？** 缓存满了要腾地方，删掉"最久没被访问"的数据。热门帖子会一直被访问，不会被删；冷门帖子久了自动淘汰。
>
> **方案A的架构：**
>
> ```
> 用户请求帖子详情
>     ↓
> 先查 Redis缓存（按postId）
>     ↓
> 命中 → 直接返回（超快！）
> 未命中 → 查DynamoDB → 存入缓存 → 返回
> ```
>
> **问题**：分布式缓存按postId分片（Sharding），爆款帖子还是只在一个缓存节点上，热键问题没根本解决！
>
> > 💡 **什么是分片（Sharding）？** 把数据分散存在多台机器上，每台机器负责一部分数据。比如postId以A-M开头的存第1台，N-Z开头的存第2台。
>
> ------
>
> #### ✅✅ 方案B：复制缓存（Replicated Cache，**最佳方案**）
>
> **核心思路：每个缓存节点存所有热门帖子，而不是分片存。**
>
> ```
>                     ┌─→ 缓存节点1（存全量热门帖子）
> 请求 → 负载均衡器 ─→ ├─→ 缓存节点2（存全量热门帖子）
>                     └─→ 缓存节点3（存全量热门帖子）
> ```
>
> **对比两种方案：**
>
> |              | 分片缓存（Sharded）        | 复制缓存（Replicated）                    |
> | ------------ | -------------------------- | ----------------------------------------- |
> | 热键问题     | ❌ 爆款帖子还是打同一个节点 | ✅ 流量均摊到所有节点                      |
> | 总存储量     | 大（每个帖子只存1份）      | 小（每个帖子存N份，N=节点数）             |
> | 缓存未命中时 | 只需1次数据库请求          | 最多N次数据库请求（第一次各节点都未命中） |
> | 适合场景     | 数据量大但访问均匀         | 数据量小但极度热点                        |
>
> > N次数据库请求听起来多，但实际上N只是几十个节点，比起百万次请求已经好了10万倍！
>
> **对于帖子编辑（不是发帖，是修改已有帖子）：**
>
> - 直接让这个postId的缓存失效（cache invalidation）
> - 下次请求时重新从数据库加载
>
> ------
>
> ## 第六步：最终架构总结
>
> ```
> 用户
>   ↓
> API网关 / 负载均衡器
>   ↓              ↓              ↓
> Post Service  Follow Service  Feed Service
>   ↓                              ↓         ↓
> SQS队列          PrecomputedFeed表  复制Redis缓存
>   ↓                              ↓         ↓
> Feed Workers ──────────────→  DynamoDB（Posts + Follow + PrecomputedFeed）
> ```
>
> **数据流总结：**
>
> ```
> 【发帖流程】
> 用户发帖 → Post Service → 存Posts表 + 推消息到SQS
> 后台 Feed Worker 消费SQS → 更新粉丝的PrecomputedFeed表
> 
> 【读Feed流程】
> 用户请求 → Feed Service
>   → 读PrecomputedFeed（普通用户的预计算帖子）
>   → 实时查大V最新帖子（少量）
>   → 合并排序
>   → 查帖子详情时先查Redis缓存，未命中再查DynamoDB
>   → 返回
> ```

> ## 📌 一句话记忆卡片
>
> | 考点                 | 一句话记忆                                |
> | -------------------- | ----------------------------------------- |
> | Fan-out on Read      | 读时拼装Feed，关注多的人很慢              |
> | Fan-out on Write     | 发帖时推送，读时直接取，但大V写入太多     |
> | 混合策略             | 普通用户写时扇出，大V读时扇出，读取时合并 |
> | SQS异步队列          | 发帖人不用等推送完成，后台Worker慢慢处理  |
> | PrecomputedFeed      | 预计算好的Feed表，只存最近200条postId     |
> | cursor分页           | 时间戳作书签，比页码更稳定                |
> | 复制缓存 vs 分片缓存 | 复制缓存解决热键，每个节点都有爆款帖子    |
> | PUT的幂等性          | 多次执行结果一样，防止重复关注            |
> | GSI                  | 同一张表的另一套索引，支持不同查询方向    |
> | 1分钟最终一致性      | 允许一点延迟，换来系统可以异步处理        |
>
> ------
>
> > 🎯 **这道题的加分点**：主动提到"可以问面试官是否可以限制最大关注数"（Facebook限制5000好友），并说明生产系统中用限制来简化极端情况是完全合理的工程决策。这体现了工程成熟度。

## Rate Limiter 

> ------
>
> ## 🧠 先理解：Rate Limiter 是什么？
>
> **Rate Limiter（限流器）** = API 的保安。
>
> 你的服务器就像一家餐厅，每分钟只能服务 100 个人。如果来了 200 个人，保安（Rate Limiter）就会挡住后 100 个，说"对不起，人满了，等一会儿再来"。
>
> 技术上：每个用户每分钟最多发 100 个请求 → 超过了就返回 **HTTP 429 Too Many Requests**。
>
> ------
>
> ## 第一步：需求确认（Requirements）
>
> > 面试一开始，**先问需求，不要直接设计**。这显示你有工程思维。
>
> ### 你要说的话：
>
> "在开始设计之前，我想先确认几个问题……"
>
> ### 功能需求（Functional Requirements）
>
> | 需求         | 说明                                                     |
> | ------------ | -------------------------------------------------------- |
> | 识别客户端   | 通过 User ID / IP / API Key 区分谁在发请求               |
> | 限制请求     | 根据可配置的规则限流，比如"每分钟 100 次"                |
> | 拒绝超限请求 | 返回 HTTP 429，附带有用的 headers 告诉客户端啥时候能重试 |
>
> ### 非功能需求（Non-Functional Requirements）
>
> | 需求          | 说明                                  |
> | ------------- | ------------------------------------- |
> | 低延迟        | 每次限流检查 < 10ms，不能拖慢正常请求 |
> | 高可用        | 限流服务挂了不能让整个 API 崩         |
> | 高并发        | 支持 **100万请求/秒**，1亿日活用户    |
> | 最终一致性 OK | 不同节点之间允许短暂的限流数据不同步  |
>
> ------
>
> ## 第二步：核心实体（Core Entities）
>
> > 面试中快速过，1分钟搞定
>
> ```
> Rule（规则）     → 定义"谁" 在"什么接口"能发"多少"请求
>                    例：普通用户 1000次/小时，搜索接口 10次/分钟
> 
> Client（客户端） → 被限流的对象，可以是 User ID / IP / API Key
> 
> Request（请求）  → 进来的 HTTP 请求，携带客户端身份和时间戳
> ```
>
> **三者关系**：Request 进来 → 识别 Client → 查找适用的 Rule → 决定放行还是拒绝
>
> ------
>
> ## 第三步：系统接口（System Interface）
>
> > 面试中一句话定义核心 API
>
> ```
> isRequestAllowed(clientId, ruleId) 
> → { allowed: boolean, remaining: number, resetTime: timestamp }
> ```
>
> 这个返回值对应响应头：
>
> - `X-RateLimit-Remaining: 5`（还剩几次）
> - `X-RateLimit-Reset: 1700000000`（啥时候重置）
> - `Retry-After: 60`（多少秒后重试）
>
> ------
>
> ## 第四步：高层设计（High-Level Design）
>
> ### 🏠 先解决：Rate Limiter 放在哪里？
>
> > 这是面试第一个关键决策点，你要对比三个方案
>
> ------
>
> #### 方案一：In-Process（每个服务器自己限流）
>
> **直觉理解**：每个服务器自己记"这个用户来了多少次"。
>
> **问题**：你有 5 台服务器，用户发了 100 个请求，每台看到 20 个，每台都觉得"没超限"，放行。但实际上用户发了 100 个！
>
> ❌ 无法在多台服务器之间共享状态，**根本不能用于分布式系统**。
>
> ------
>
> #### 方案二：Dedicated Service（独立的限流微服务）
>
> **直觉理解**：专门开一家"审批公司"，所有服务器的请求都先去审批公司问"这个用户能过吗"。
>
> **优点**：全局状态统一，逻辑灵活
>
> **问题**：每个请求都要额外发一次网络请求给审批公司，**增加延迟**。还有单点故障风险。
>
> ⚠️ 可用，但有成本。
>
> ------
>
> #### 方案三：API Gateway（在网关层限流）✅ 我们选这个
>
> **直觉理解**：在大门口放保安，进门前就检查，不让超限的人进来。应用服务器根本不用管。
>
> **优点**：
>
> - 请求进不来就被拒，不消耗后端资源
> - 所有流量天然经过网关，天然集中
> - 无需额外网络跳转（相比方案二）
>
> **限制**：只能看到 HTTP 请求里的信息（headers、IP 等），看不到业务逻辑
>
> ✅ **这是生产环境最常用方案，我们选这个。**
>
> ------
>
> ### 🪪 如何识别客户端？
>
> | 标识符     | 使用场景        | 来源                              |
> | ---------- | --------------- | --------------------------------- |
> | User ID    | 已登录用户      | JWT Token（Authorization header） |
> | IP Address | 未登录/公开 API | X-Forwarded-For header            |
> | API Key    | 开发者 API      | X-API-Key header                  |
>
> **实际系统会叠加多层规则**，取最严格的：
>
> ```
> Alice 个人限制: 1000次/小时  (还剩 500次)
> Alice 的 IP 限制: 100次/分钟  (已用完!)
> → 结果：Alice 被拦截（因为 IP 这条触发了）
> ```
>
> ------
>
> ### ⚙️ 核心算法：选哪种限流算法？
>
> > 这是面试最核心的技术讨论，你要能讲清楚所有算法再说选哪个
>
> ------
>
> #### 算法一：Fixed Window Counter（固定窗口计数器）
>
> **类比**：时钟整点重置的停车计时器。12:00到12:59 是一个格，最多停 100 辆车。
>
> **实现**：用 hash 表，`用户ID:时间窗口开始时间 → 计数器`
>
> ```
> alice:12:00:00 = 100  ← 满了
> alice:12:01:00 = 5    ← 新格子
> ```
>
> **致命缺陷**：边界突破攻击！
>
> ```
> 12:00:59 发 100 个请求 ✅（第一个格子最后一秒）
> 12:01:00 发 100 个请求 ✅（第二个格子第一秒）
> → 2秒内发了 200 个请求，但都没被拦截！
> ```
>
> ❌ 简单但有安全漏洞
>
> ------
>
> #### 算法二：Sliding Window Log（滑动窗口日志）
>
> **类比**：记录每次停车的精确时间戳，检查时看"过去 60 分钟内"有几条记录。
>
> **优点**：完全精确，无边界问题
>
> **问题**：每个用户需要存储所有请求的时间戳。100万用户 × 1000次/小时 = **存储爆炸**
>
> ❌ 内存太贵，不适合大规模
>
> ------
>
> #### 算法三：Sliding Window Counter（滑动窗口计数器）
>
> **类比**：用"当前格子"和"上一个格子"的数据，按比例估算过去 60 秒的请求数。
>
> **公式**：`估算请求数 = 上个窗口计数 × (1 - 已过去的比例) + 当前窗口计数`
>
> **例子**：
>
> ```
> 当前时刻：12:01:45（在12:01:00-12:01:59窗口内，已过75%）
> 上个窗口(12:00) 有 80 个请求
> 当前窗口(12:01) 有 30 个请求
> 
> 估算 = 80 × (1 - 0.75) + 30 = 20 + 30 = 50个请求
> ```
>
> ✅ 内存省（只需两个计数器），近似精确，是很好的折中
>
> ------
>
> #### 算法四：Token Bucket（令牌桶）✅ 我们选这个
>
> **类比（关键！记住这个）**：
>
> > 想象一个桶，系统不断往里加水（令牌）。你每次请求就舀一瓢水。桶满了多余的水溢出（令牌不超过上限）。桶空了就不能舀水（请求被拒）。
>
> | 概念                    | 类比       | 作用              |
> | ----------------------- | ---------- | ----------------- |
> | Bucket（桶）            | 水桶       | 允许短暂突发      |
> | Token（令牌）           | 水         | 每次请求消耗 1 个 |
> | Refill Rate（补充速率） | 水龙头速度 | 控制长期速率      |
> | Capacity（容量）        | 桶的大小   | 控制最大突发量    |
>
> **例子**：
>
> ```
> 桶容量: 100 tokens（最多存 100 个）
> 补充速率: 10 tokens/分钟
> 
> 用户可以先冲一波发 100 个请求（突发）
> 然后每分钟最多发 10 个（平稳）
> ```
>
> **每个用户只需存储**：`(当前token数, 上次补充时间)` → 内存极省
>
> ✅ **Stripe 等大公司都用这个。支持突发流量，内存高效，实现简洁。我们选它。**
>
> ------
>
> ### 🗄️ 状态存在哪里？引入 Redis
>
> > 这是高频面试考点，必须讲清楚
>
> **问题**：我们有多个 API Gateway 实例，每个用户的令牌桶状态必须共享，不然每个 Gateway 各自计数，结果不准。
>
> **答案：Redis**
>
> **Redis 是什么（给小白）**：Redis 就是一个超快的、存在内存里的"共享字典"。所有 Gateway 都去这个字典里读写用户的令牌桶状态。速度 < 1ms。
>
> **操作流程**：
>
> ```
> 1. 请求到达 Gateway A，用户是 Alice
> 2. Gateway 读取 Redis: alice:bucket → { tokens: 50, last_refill: 10:00:00 }
> 3. 计算应该补充多少 tokens（根据时间差）
> 4. 原子更新 Redis：写入新的 token 数和时间戳
> 5. token > 0 → 放行；token = 0 → 返回 429
> ```
>
> **Redis 存储的数据**：
>
> ```
> alice:bucket = {
>   tokens: 50,
>   last_refill: 1700000000
> }
> ```
>
> 设置 1小时 TTL（过期自动删除），防止内存泄漏。
>
> ------
>
> ### ⚠️ 竞态条件问题及解决
>
> > 面试加分点，主动提出这个问题
>
> **问题**：两个 Gateway 同时读取 Alice 的桶，都看到 1 个 token，都认为可以放行，都让请求通过。但实际上只有 1 个 token，应该只允许 1 个请求！
>
> **解决方案：Lua 脚本**
>
> **Lua 脚本是什么**：Redis 支持在服务器端运行小程序（Lua脚本）。整个"读取 → 计算 → 写入"在 Redis 里**原子执行**，不会被其他操作打断。
>
> ```lua
> -- 伪代码
> local tokens = redis.get(key)
> local new_tokens = calculate(tokens, current_time)
> if new_tokens > 0 then
>   redis.set(key, new_tokens - 1)
>   return "ALLOW"
> else
>   return "DENY"
> end
> ```
>
> 一个原子操作 = 没有竞态条件 ✅
>
> ------
>
> ## 第五步：深度讨论（Deep Dives）
>
> > 主动引导面试官讨论这些话题，展示你考虑了扩展性
>
> ------
>
> ### Deep Dive 1：如何支撑 100万请求/秒？
>
> **问题**：单个 Redis 实例只能处理约 10万-20万次操作/秒，但我们需要处理 100万请求/秒（每次限流需要多次 Redis 操作），远超上限。
>
> **解决方案：Redis 分片（Sharding）**
>
> **分片是什么（给小白）**：把数据分散到多个 Redis 实例上，每个实例负责一部分用户。就像图书馆的书分 A-G、H-M、N-Z 三个区，找书时先去对应区。
>
> **如何分配用户到哪个分片：一致性哈希（Consistent Hashing）**
>
> **一致性哈希是什么**：对用户 ID（或 IP）做哈希运算，结果决定去哪个 Redis 实例。重要特性：同一个用户**永远**去同一个实例（保证状态一致），加减实例时只有少量用户需要迁移。
>
> ```
> hash(alice) % 10 = 3  → 去 Redis 实例 #3
> hash(bob)   % 10 = 7  → 去 Redis 实例 #7
> ```
>
> **结果**：10 个 Redis 实例 × 10万次/秒 = 100万次/秒 ✅
>
> **生产实践**：直接用 **Redis Cluster**（Redis 官方集群方案），它自动做分片，不需要你手动写哈希路由逻辑。
>
> ------
>
> ### Deep Dive 2：Redis 挂了怎么办？高可用
>
> **两种故障处理策略**：
>
> | 策略                        | 行为                     | 适用场景                       |
> | --------------------------- | ------------------------ | ------------------------------ |
> | **Fail-Open（开放失败）**   | Redis 挂了就放行所有请求 | 宁可流量泄漏，不能服务中断     |
> | **Fail-Closed（关闭失败）** | Redis 挂了就拒绝所有请求 | 宁可服务中断，不能失去限流保护 |
>
> **我们选 Fail-Closed（关闭失败）**
>
> 原因：对于社交媒体平台，Redis 经常在流量高峰时出问题。如果这时 Fail-Open 放行所有请求，海量流量涌入后端，会导致**级联崩溃**（cascade failure）——限流器故障变成全站崩溃，更糟糕。
>
> **如何减少 Redis 故障：主从复制（Master-Replica）**
>
> **主从复制是什么**：每个 Redis 分片有一个主节点（Master）负责读写，有一个或多个从节点（Replica）实时同步数据。主节点挂了，从节点自动升级为新主节点。Redis Cluster 内置这个功能，自动故障转移。
>
> ```
> Redis Shard 1: [Master] ← 同步 → [Replica]
>                 ↑ 挂了          ↑ 自动升为 Master
> ```
>
> ------
>
> ### Deep Dive 3：如何降低延迟？
>
> | 优化手段                                | 原理                                                | 效果                                 |
> | --------------------------------------- | --------------------------------------------------- | ------------------------------------ |
> | **连接池（Connection Pooling）**        | 提前建好一批 TCP 连接，请求来了直接用，不用每次握手 | 省去 TCP 握手的 20-50ms              |
> | **地理分布（Geographic Distribution）** | 在不同地区部署 Redis，用户就近访问                  | 北京用户不用连美国，延迟从 150ms→5ms |
> | **Lua 脚本合并操作**                    | 把读+算+写合并成一次 Redis 调用                     | 减少网络往返次数                     |
>
> > 面试中：重点说**连接池**和**地理分布**，其他的"如果面试官追问再展开"。
>
> ------
>
> ### Deep Dive 4：热点问题（Hot Key）
>
> **什么是热点**：某个用户/IP 产生极大量请求，导致负责它的那个 Redis 分片被压垮，而其他分片很闲。
>
> **合法来源**：企业级 API 客户、大型数据爬虫、手机 App 频繁刷新 **恶意来源**：DDoS 攻击、机器人刷接口
>
> **应对策略**：
>
> | 场景         | 策略                                                         |
> | ------------ | ------------------------------------------------------------ |
> | 合法高频用户 | 提供高级套餐（更高限额），或允许客户端自己做限流（SDK 内置） |
> | 恶意攻击     | 触发多次限流后自动封 IP（加入 blocklist），上游用 Cloudflare / AWS Shield 过滤 |
>
> ------
>
> ### Deep Dive 5：动态规则配置
>
> > 如何在不重新部署代码的情况下修改限流规则？
>
> | 方案                              | 原理                               | 更新延迟   | 复杂度 |
> | --------------------------------- | ---------------------------------- | ---------- | ------ |
> | **轮询（Poll-Based）**            | Gateway 每 30 秒去数据库拉一次规则 | 最多 30 秒 | 低     |
> | **推送（Push-Based，ZooKeeper）** | 规则变化立即推送给所有 Gateway     | 秒级       | 高     |
>
> **我们选轮询**：30 秒延迟对大多数场景够用，实现简单。只有需要毫秒级响应（比如安全事件紧急封锁）才用推送。
>
> **ZooKeeper 是什么（给小白）**：一个专门做"分布式配置同步"的系统。任何节点的配置变了，它马上通知所有连接的服务。类比：WhatsApp 群消息，有人发消息，所有人实时收到。
>
> ------
>

>
> > "通过 Redis Cluster 的一致性哈希分片，10个分片每个处理10万QPS，合计支撑100万QPS目标。"
>
> ```
> ┌─────────────────────────────────────────────────────────────────────────────────┐
> │                              CLIENTS                                            │
> │         [Mobile App]        [Web Browser]        [Third-party Developer]        │
> │         User ID in JWT      User ID in JWT        API Key in Header             │
> └───────────────────────────────────┬─────────────────────────────────────────────┘
>                                     │ HTTP Requests
>                                     ▼
> ┌─────────────────────────────────────────────────────────────────────────────────┐
> │                         DDoS PROTECTION LAYER                                   │
> │                    [Cloudflare / AWS Shield]                                    │
> │              Blocks malicious traffic before it enters system                   │
> └───────────────────────────────────┬─────────────────────────────────────────────┘
>                                     │ Filtered Traffic
>                                     ▼
> ┌─────────────────────────────────────────────────────────────────────────────────┐
> │                            LOAD BALANCER                                        │
> │                  Distributes traffic across API Gateways                        │
> └──────────┬─────────────────────┬──────────────────────┬────────────────────────┘
>            │                     │                      │
>            ▼                     ▼                      ▼
> ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐

> ## 每个组件详解
>
> ### 1. Clients（客户端）
>
> 系统的使用者，分三类：手机 App 用户、网页用户、第三方开发者。
>
> **关键点**：每类客户端携带的身份标识不同。已登录用户在请求的 Authorization Header 里带着 JWT Token（一种加密的身份证明），里面包含 User ID。第三方开发者用 API Key 标识身份。
>
> 这就是我们后续识别"谁在发请求"的信息来源。
>
> ------
>
> ### 2. DDoS Protection Layer（DDoS 防护层）
>
> **DDoS 是什么**：攻击者用大量机器同时向你的服务器发请求，把服务器打垮。
>
> **Cloudflare / AWS Shield** 是专业的流量清洗服务。它们在请求进入你的系统之前，就识别并过滤掉明显的恶意流量（比如某个 IP 每秒发 10 万个请求）。
>
> 放在最前面是因为：如果不挡，恶意流量连 Rate Limiter 都到不了，直接把 API Gateway 压垮了。
>
> ------
>
> ### 3. Load Balancer（负载均衡器）
>
> **是什么**：把进来的请求分发给多台 API Gateway，防止某一台被压垮。
>
> **类比**：超市收银台。顾客不是全挤到同一个收银员那里，而是被引导到不同的队列。
>
> 它做的事很简单：收到请求 → 选一台 API Gateway → 转发过去。常用策略是轮询（Round Robin），依次分给每台 Gateway。
>
> ------
>
> ### 4. API Gateway（API 网关）
>
> 这是整个架构的**核心决策节点**，每个请求都必须经过这里。每台 Gateway 做四件事：
>
> **第一步：提取客户端身份** 从请求头里读 User ID（解析 JWT Token）、IP 地址（X-Forwarded-For Header）、或 API Key（X-API-Key Header）。
>
> **第二步：本地缓存查规则** 不是每次都去数据库查"这个用户限制多少次"，而是每 30 秒把规则拉下来缓存在本地内存里。查起来速度极快。
>
> **第三步：调用 Redis 执行 Lua 脚本** 用客户端 ID 算出去哪个 Redis Shard，然后在那个 Shard 上运行 Lua 脚本，原子性地完成"读 token → 计算补充 → 写回 → 返回决策"。
>
> **第四步：放行或拒绝** 根据 Redis 返回的结果，允许请求继续走，或者立即返回 429。
>
> ------
>
> ### 5. Consistent Hashing（一致性哈希）
>
> **解决的问题**：有多个 Redis Shard，请求来了该去哪个？
>
> **规则**：对客户端 ID 做哈希运算，结果决定去哪个 Shard。
>
> ```
> hash("alice") → 去 Shard #1
> hash("bob")   → 去 Shard #2
> ```
>
> **为什么不能随机分配**：Alice 的 token 桶数据只存在一个地方。如果第一次请求去了 Shard #1，第二次请求去了 Shard #2，第二次看不到第一次消耗的 token，限流就失效了。
>
> **一致性的意义**：同一个用户，无论哪台 Gateway 处理它的请求，哈希值永远相同，永远去同一个 Shard。这保证了状态的正确性。
>
> ------
>
> ### 6. Redis Cluster（Redis 集群）
>
> 这是**全局状态存储**，所有 Gateway 共享这里的数据。
>
> **存储的数据格式（Token Bucket 状态）**：
>
> ```
> alice:bucket = {
>   tokens: 50,
>   last_refill: 1700000000
> }
> ```
>
> 就这两个值，每个用户只占极小的内存。
>
> **Lua Script（Lua 脚本）的作用**：把"读取 → 计算 → 写入"三个步骤合并成一个原子操作，防止两个 Gateway 同时操作同一个用户的桶产生竞态条件。脚本在 Redis 服务器上执行，执行期间没有任何其他操作可以插入。
>
> **Blocklist（封禁名单）**：Redis 里还存着一个黑名单，记录被封禁的 IP 或 User ID。Gateway 在检查 token 之前先查这个名单，黑名单里的直接拒绝，不用跑限流逻辑。
>
> ------
>
> ### 7. Master-Replica（主从复制）
>
> 每个 Redis Shard 不只有一台机器，而是一主一从（或一主多从）。
>
> **Master（主节点）**：负责处理所有读写请求，是数据的权威来源。
>
> **Replica（从节点）**：实时同步 Master 的数据。平时是备用，Master 挂了就自动升级为新的 Master，整个过程自动完成，不需要人工干预。
>
> 这就是为什么我们敢说"Redis 挂了 Fail-Closed"——实际上 Redis 很少挂，因为有主从保护。
>
> ------
>
> ### 8. Decision Outcome（决策结果）
>
> **ALLOW（放行）**：请求静默转发给后端，用户无感知。
>
> **DENY（拒绝）**：立即返回 HTTP 429，不转发给后端。响应包含四个关键 Header：
>
> - `X-RateLimit-Limit: 100` → 你的总配额是多少
> - `X-RateLimit-Remaining: 0` → 还剩多少（这里已经用完了）
> - `X-RateLimit-Reset: 1700000000` → 什么时间点（Unix 时间戳）配额重置
> - `Retry-After: 60` → 再等多少秒可以重试
>
> 这些 Header 让客户端知道等多久，而不是盲目重试狂刷。
>
> ------
>
> ### 9. Backend Services（后端服务）
>
> 真正的业务逻辑在这里，比如发帖、看时间线、搜索等。
>
> **它们完全不需要知道 Rate Limiter 的存在**。到达这里的请求都是"已经审查合格的"，后端专心处理业务即可。这就是 API Gateway 方案的优雅之处。
>
> ------
>
> ### 10. Rule Configuration System（规则配置系统）
>
> 这是一个独立于请求处理路径之外的管理系统。
>
> **Rule Database（规则数据库）**：存储所有限流规则，比如"普通用户 1000次/小时"、"搜索接口 10次/分钟"。用普通关系型数据库（如 PostgreSQL）就够了，因为规则变化频率很低。
>
> **轮询机制**：每台 API Gateway 每 30 秒去数据库拉一次最新规则，存在本地内存缓存里。这样查规则的速度极快（本地内存），代价是规则更新最多有 30 秒延迟。
>
> **Admin Panel（管理面板）**：运维人员修改规则的界面，比如临时给某个大客户提高配额，或者紧急降低某个接口的限制。
