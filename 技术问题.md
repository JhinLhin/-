**第一个讲：CIM Monitor Dashboard（Tab A + Tab B 合在一起讲）**——这是最强的项目，因为它天然有两个子系统可以对比：Tab A 用实时推送，Tab B 用轮询，你能清楚解释为什么同一个产品里对不同数据源选了不同架构。VPC 安全组互引、pg8000 跨平台踩坑、NONE resolver 做纯广播这三个点分别展示了安全设计、debugging 能力、和系统设计理解。

**第二个讲：ECS auto stop**——方案选型对比最丰富（四种方案逐个否决），Tagging API 做动态发现是亮点。

**第三个讲：EC2 auto stop**——KMS 加密 EBS 卷的排障故事最精彩，two-role separation 展示 least privilege 思维。



## CIM Collector Real-Time Monitoring Dashboard 

> ------
>
> ## 一句话 Elevator Pitch
>
> > "I designed and built a real-time monitoring dashboard from scratch — NiFi ingests collector heartbeat data into DynamoDB, DynamoDB Streams triggers a Lambda that pushes events through AppSync GraphQL subscriptions to a React frontend, giving operators instant visibility into collector health with zero polling."
>
> ------
>
> ## STAR 框架回答
>
> ### Situation
>
> - 团队使用 Apache NiFi 做数据管道，有多个 data collector 在不同 tenant 上运行
> - 当 collector 停止发送数据（inactive）或恢复（restored）时，**没有实时告警**——运维只能靠手动检查 NiFi 流程或看日志才能发现问题
> - 发现问题的平均时间太长，影响数据质量和 SLA
>
> ### Task
>
> - 构建一个**实时监控 dashboard**：collector 状态变化（active / inactive / restored）必须秒级推送到前端
> - 页面打开时要能看到历史状态（不能只等实时推送）
> - 数据自动过期清理（不能无限膨胀）
> - Prod 环境必须要求用户登录（Cognito），Dev 可以 guest access
>
> ### Action
>
> **端到端架构设计**
>
> ```
> NiFi (MonitorActivity processor)
>   → DynamoDB (composite key: collector_id + timestamp, TTL 自动过期)
>     → DynamoDB Streams (capture INSERT/MODIFY)
>       → Lambda (SigV4 签名调 AppSync)
>         → AppSync Mutation (NONE resolver, 不写库, 纯广播)
>           → GraphQL Subscription (WebSocket)
>             → React Frontend (Amplify + Cognito IAM auth)
> ```
>
> **两条独立数据路径（面试关键点）**
>
> | 路径     | 用途             | 链路                                                         | Resolver 类型                  |
> | -------- | ---------------- | ------------------------------------------------------------ | ------------------------------ |
> | 实时推送 | 状态变更秒级通知 | DynamoDB Streams → Lambda → AppSync Mutation → Subscription → 前端 | **NONE**（不碰数据库，纯广播） |
> | 历史查询 | 页面加载时拉快照 | 前端 → AppSync Query → DynamoDB → 返回数据                   | **DynamoDB**（直接读表）       |
>
> 前端用 **query-then-subscribe** 模式：先走第二条路拉历史，再走第一条路接实时更新。
>
> **DynamoDB 设计**
>
> - Composite Primary Key：`collector_id` (Partition Key) + `timestamp` (Sort Key)
> - 同一个 collector 可以有多条时间序列记录
> - TTL attribute (`ttl`)：每条记录写入时计算过期时间（epoch + 30 天），DynamoDB 自动删除过期数据
> - Stream 模式选 **New Image**：每次写入推送完整新记录给 Lambda
>
> **NiFi 数据采集层**
>
> - 使用 `MonitorActivity` processor 检测数据流活跃度
> - 三个输出关系 → 三个 `UpdateAttribute` processor 设置 status 和 TTL → `ReplaceText` 拼 JSON → `PutDynamoDBRecord` 写入
> - inactive / restored / active 三种状态各自独立链路，互不干扰
>
> **Lambda 设计（胶水层）**
>
> - 从 DynamoDB Stream 的 `NewImage` 提取字段（处理 DynamoDB 的特殊类型包装 `{S: "value"}` / `{N: "123"}`）
> - 只处理 INSERT 和 MODIFY，跳过 REMOVE（TTL 删除产生的事件）
> - 用 SigV4 签名调 AppSync（IAM auth），不用 API Key
> - 在 Event Source Mapping 层加了 **Filter Criteria**（`eventName: [INSERT, MODIFY]`），REMOVE 事件在 AWS 基础设施层就被过滤，Lambda 根本不会被唤醒
>
> **AppSync GraphQL 设计**
>
> - Primary auth mode: **AWS_IAM**（不用 API Key，更安全）
> - Mutation `pushCollectorEvent`：绑 **NONE resolver**——不写库，只触发 subscription 广播
> - Query `listCollectorEvents`：绑 **DynamoDB resolver**——Scan 表返回最近 50 条
> - Subscription `onCollectorEvent`：通过 `@aws_subscribe(mutations: ["pushCollectorEvent"])` 自动绑定
>
> **认证设计（Dev vs Prod）**
>
> | 环境 | 认证方式                          | Guest Access |
> | ---- | --------------------------------- | ------------ |
> | Dev  | Cognito Identity Pool (guest)     | ✅ 允许       |
> | Prod | Cognito User Pool + Identity Pool | ❌ 必须登录   |
>
> Prod 链路：用户输入邮箱密码 → User Pool 验证 → Cognito token → Identity Pool 换成 IAM 临时凭证 → 浏览器用凭证调 AppSync。IAM policy 精确到 field 级别，只允许 `listCollectorEvents` query 和 `onCollectorEvent` subscription。
>
> ### Result
>
> - Collector 状态变化从"发现需要几小时"变为**秒级推送到 dashboard**
> - 页面打开即有历史数据，实时更新无需轮询
> - 数据自动过期清理，表不会无限增长
> - 零轮询架构：全链路事件驱动，不浪费计算资源
>
> ------
>
> ## 高频追问 & 回答要点
>
> ### Q1: "Walk me through the real-time data flow end to end."
>
> > NiFi 的 MonitorActivity processor 检测到 collector 状态变化，写入 DynamoDB。DynamoDB Streams 捕获这个 INSERT 事件，触发 Lambda。Lambda 从 Stream record 的 NewImage 里提取 collector_id、timestamp、status、message，然后用执行角色的 IAM 临时凭证做 SigV4 签名，POST 一个 GraphQL mutation 到 AppSync。AppSync 的 mutation 绑的是 NONE resolver——它不写任何数据库，只是把 payload 原样广播给所有订阅了 `onCollectorEvent` 的 WebSocket 客户端。前端收到数据后更新两张表：Recent Events 和 Collector Status Overview。
>
> ### Q2: "Why NONE resolver instead of DynamoDB resolver on the mutation?"
>
> > 因为数据**已经在 NiFi 写入 DynamoDB 了**。如果 mutation 再用 DynamoDB resolver 写一次，就会重复写入。NONE resolver 的意思是"AppSync 不碰任何数据源，只做消息中转"——Lambda 传什么，subscription 就广播什么。这样 AppSync 在这条实时推送路径上是纯广播层，没有副作用。
>
> ### Q3: "Why not just have the frontend poll DynamoDB?"
>
> > 三个原因。第一，轮询有延迟——即使每 5 秒 poll 一次，最坏情况也要等 5 秒才能发现 collector 挂了，而 WebSocket 推送是亚秒级。第二，轮询浪费资源——30 个 collector 如果没有状态变化，99% 的 poll 请求返回的是重复数据，白白消耗 DynamoDB 读容量。第三，轮询不 scale——如果有 10 个运维同时看 dashboard，就是 10 倍的 poll 请求；WebSocket 模式下 AppSync 只广播一次，所有客户端都收到。
>
> ### Q4: "Explain the two data paths and why you need both."
>
> > 实时推送路径（Streams → Lambda → AppSync subscription）只能推送"从你建立 WebSocket 连接之后"的新事件。如果用户刷新页面或者刚打开 dashboard，他们看不到任何历史数据。所以我加了第二条路径：前端通过 AppSync Query 直接 Scan DynamoDB，拉最近 50 条记录作为初始快照。前端代码的模式是 query-then-subscribe——先加载快照填充表格，再建立 subscription 接收实时更新。两条路径的数据源是同一个 DynamoDB 表，但经过的链路完全不同。
>
> ### Q5: "How does SigV4 signing work in your Lambda?"
>
> > Lambda 执行角色自带临时 IAM 凭证（access key + secret key + session token）。我用 botocore 的 `SigV4Auth` 对 HTTP 请求做签名——它会计算 request body 的 hash，加上 host、date 等 headers，用 secret key 做 HMAC 计算，生成 `Authorization`、`X-Amz-Date`、`X-Amz-Security-Token` 三个头。AppSync 收到请求后用同样的算法验证签名，并检查 IAM policy 里是否允许 `appsync:GraphQL` 操作。整个过程不传输密码，签名包含 body hash 所以防篡改。
>
> ### Q6: "What was your hardest debugging moment?"
>
> > AppSync 返回 401 Unauthorized。Lambda 代码逻辑没问题，签名也正确。排查了一个小时后发现：AppSync 的 **API ID** 和 **GraphQL endpoint URL 前缀**是两个不同的值。我在 IAM policy 的 Resource ARN 里用了 URL 前缀（`oakvnt7ly...`），但 IAM 评估需要的是 API ID（`5in6rpie...`）。一字之差导致 IAM 找不到匹配的 resource，所以返回 401。修复后 Lambda 立刻通了。教训是：AppSync 的 ARN 必须用 Settings 页面里的 API ID，不能从 endpoint URL 猜。
>
> ### Q7: "How do you handle data cleanup / TTL?"
>
> > DynamoDB 的 TTL 是 per-item 的，不是全局统一时间。NiFi 写入每条记录时计算 `ttl = now + 30 天`（epoch 秒），DynamoDB 后台自动删除过期数据。TTL 删除会产生 REMOVE 事件到 Stream，但我在 Lambda 的 Event Source Mapping 上配了 Filter Criteria 只接收 INSERT 和 MODIFY，所以 REMOVE 事件在 AWS 基础设施层就被丢弃，Lambda 根本不会被唤醒，不浪费钱。前端不需要实时感知 TTL 删除——运维关心的是状态变化，不是数据清理。页面刷新时重新 Scan 自然只拿到未过期的数据。
>
> ### Q8: "How does authentication differ between dev and prod?"
>
> > Dev 用 Cognito Identity Pool 的 guest access——不需要登录，浏览器直接拿到 IAM 临时凭证。适合快速验证。Prod 加了 Cognito User Pool——用户必须用邮箱密码登录，关闭了 self-registration，只有管理员手动创建的账号才能访问。登录后 Cognito token 通过 Identity Pool 换成 IAM 凭证，前端再用这个凭证调 AppSync。IAM policy 精确到 GraphQL field 级别：只允许 `listCollectorEvents`（query）和 `onCollectorEvent`（subscription），mutation 等其他操作全部被拒绝。
>
> ### Q9: "Why didn't you use EventBridge to call AppSync directly instead of Lambda?"
>
> > 我之前的方案确实考虑过 EventBridge 直接调 AppSync target。但 EventBridge 作为 AWS 服务调 AppSync 走的是不同的认证路径，而且 DynamoDB Stream 的 record 格式（带类型包装的 `{S: "value"}`）需要转换成 AppSync mutation 的参数格式。Lambda 让我有完全的控制权：字段提取、类型转换、错误处理、日志记录都很直接。60 行 Python，不需要额外依赖，维护成本很低。
>

## CIM Export Job Monitoring

> ------
>
> ## 一句话 Elevator Pitch
>
> > "I added a failed-export-job monitoring tab to an existing real-time dashboard — Lambda queries PostgreSQL over a private VPC link, exposes results through API Gateway REST, and the React frontend polls every 30 seconds. The whole thing cost under $1/month and I secured the database connection with VPC + security-group-referencing instead of opening it to the public internet."
>
> ------
>
> ## STAR 框架回答
>
> ### Situation
>
> - 团队已有一个实时监控 dashboard（Tab A）监控 NiFi collector 健康状态
> - 但**导出作业（export jobs）**的失败记录没有可视化——运维只能手动登录 PostgreSQL 查 `export_job` 表（38M 行），跑 SQL 看哪些 job 失败了
> - 数据库跑在 EC2 上，前面有 PgBouncer（端口 6432），数据源是 Citus 分布式 PostgreSQL
>
> ### Task
>
> - 在现有 dashboard 上新增 **Tab B**，展示最近 60 天内所有非成功（status ≠ 7）的 export job
> - 不能动 Tab A 的任何后端资源
> - 数据库连接必须安全（不能把 PostgreSQL 暴露到公网）
> - 成本要低（dev 环境，不想为监控花太多钱）
>
> ### Action
>
> **架构选型：Lambda + API Gateway REST + 前端轮询**
>
> 我评估了两种方案后选了轮询：
>
> | 方案                                                         | 否决原因                                                     |
> | ------------------------------------------------------------ | ------------------------------------------------------------ |
> | 复制 Tab A 的模式（DynamoDB + Streams + AppSync subscription） | PostgreSQL 的变更不会自动产生 stream；需要额外同步层把 PG 数据写入 DynamoDB；记录"变好了"（从失败变成功）时需要删除逻辑；组件更多，成本约 $2+/月 |
> | **Lambda + API Gateway + 30s 轮询** ✅                        | SQL WHERE 直接在源头过滤；记录变 valid 后下次全量查询自然消失，不需要删除逻辑；3 个组件，成本约 $0.40/月 |
>
> **关键设计决策：pg8000 替代 psycopg2**
>
> Lambda Layer 需要 PostgreSQL 驱动。我在 Windows 开发环境踩了一个坑：
>
> | 驱动            | 问题                                                         |
> | --------------- | ------------------------------------------------------------ |
> | psycopg2        | 需要编译 C 库（.so 文件），Windows 打包的 Layer 在 Lambda Linux 上运行报错 |
> | psycopg2-binary | Windows pip 下载的是 Windows 二进制，Lambda 是 Linux，报 `os.add_dll_directory` 错误 |
> | aws-psycopg2    | 仍有 `_psycopg.so` 依赖问题                                  |
> | **pg8000** ✅    | 纯 Python 实现，零 C 依赖，Windows pip install 的包直接能在 Lambda Linux 上跑 |
>
> **VPC 安全设计（最有深度的部分）**
>
> 初始方案是 EC2 安全组开 `0.0.0.0/0:6432`，但这把数据库暴露给了整个互联网。我改成了正规方案：
>
> ```
> Lambda (sg-lambda) ──私网──► EC2 PostgreSQL (sg-ec2, port 6432)
>          ↑ 同一个 VPC，走内网 IP，不经公网
> ```
>
> 具体步骤：
>
> 1. 创建 Lambda 专用安全组（`sg-lambda`），inbound 为空（Lambda 只出不进）
> 2. EC2 安全组加规则：`Custom TCP 6432, Source = sg-lambda`（安全组互引）
> 3. Lambda IAM role 加 `AWSLambdaVPCAccessExecutionRole`（ENI 管理权限）
> 4. Lambda 配置进入 EC2 所在的 VPC
> 5. `DB_HOST` 从公网 DNS 改为 EC2 私有 IP（`10.0.1.214`）
>
> 这样只有挂了 `sg-lambda` 的资源才能访问 6432 端口，数据库对公网完全不可见。
>
> **API Gateway 层**
>
> - REST API，GET `/invalid-records`
> - Lambda proxy integration（API Gateway 直接透传 Lambda 返回）
> - CORS `Access-Control-Allow-Origin: *`（后续可收紧为 Amplify 域名）
> - 部署到 `prod` stage
>
> **前端 Tab B**
>
> - `useEffect` + `setInterval` 每 30 秒轮询 API Gateway
> - 显示倒计时 + 手动 Refresh 按钮
> - Summary Cards：Total / Connection Failed (红) / Schedule Error (黄) / Other
> - 搜索过滤：Job ID、User、Tenant、Error Message
> - 状态色标：status 4 = 红，status 8 = 琥珀
> - 复用 Tab A 的 Philips 品牌设计系统和 Cognito 认证
>
> **认证复用**
>
> Tab B 不需要单独的认证——API Gateway 的 CORS 头允许前端直接 fetch，而前端本身已经被 Cognito `<Authenticator>` 保护。未登录用户看到的是 Cognito 登录页，根本到不了 Tab B 的代码。
>
> ### Result
>
> - 运维从"手动登 DB 跑 SQL"变成"打开 dashboard 看一眼"
> - 30 秒自动刷新，失败的 job 一目了然
> - 数据库零公网暴露，安全组互引确保只有 Lambda 能连
> - 整套方案月成本约 $0.40（Lambda 调用 + API Gateway 请求）
> - 部署到 Amplify Hosting，手动 drag-and-drop（公司 GitHub 组织限制了第三方 app 安装）
>

>
> ## 高频追问 & 回答要点
>
> ### Q1: "Why polling instead of real-time push like Tab A?"
>
> > Tab A 的数据源是 DynamoDB，天然有 Streams（CDC）可以触发 Lambda 做实时推送。Tab B 的数据源是 PostgreSQL，没有原生的 change stream。要做实时推送就得额外建一个同步层（比如 Debezium 或定时写 DynamoDB），增加了两三个组件。而 export job 的失败不是秒级紧急事件——运维看到 30 秒延迟完全可以接受。轮询方案只需要 3 个组件（Lambda + API Gateway + 前端 setInterval），成本低一个数量级，复杂度也低。
>
> ### Q2: "Why pg8000 instead of psycopg2?"
>
> > 这是一个 cross-platform Lambda Layer 的坑。psycopg2 底层是 C 扩展，需要编译成 `.so` 文件。我的开发环境是 Windows，`pip install psycopg2-binary` 下载的是 Windows 二进制，上传到 Lambda（Linux）后跑不了。`aws-psycopg2` 也有类似的 `_psycopg.so` 依赖问题。pg8000 是纯 Python 实现，零 C 依赖，Windows 上 pip install 的包直接能在 Lambda Linux 运行时上跑，完美解决了跨平台打包问题。
>
> ### Q3: "Walk me through the VPC security design."
>
> > 最初我直接开了 EC2 安全组的 `0.0.0.0/0:6432` 来让 Lambda 连接。但这意味着任何人知道 IP 和端口就能尝试连数据库。正规做法是把 Lambda 放进 EC2 所在的 VPC，用安全组互引来限制访问。
> >
> > 具体：我创建了一个 Lambda 专用安全组（inbound 为空），然后在 EC2 安全组里加了一条规则——6432 端口只允许来自 Lambda 安全组的流量。这样只有挂了这个安全组的 ENI 才能访问数据库，对公网完全不可见。
> >
> > Lambda 进 VPC 后还有一个关键点：`DB_HOST` 必须改成 EC2 的私有 IP，因为 Lambda 现在走的是 VPC 内网，公网 DNS 解析出来的地址不在同一网络路径上。
>
> ### Q4: "Lambda in VPC — doesn't that break API Gateway integration?"
>
> > 不会。API Gateway 调用 Lambda 走的是 AWS 内部控制平面，不走公网也不走 VPC 网络。无论 Lambda 在不在 VPC 里，API Gateway 都能正常触发它。
> >
> > 调用链是：`React → API Gateway → (AWS 内部) → Lambda (VPC 内) → (私网) → EC2 PostgreSQL`。每一段都通，不需要 NAT Gateway。
> >
> > 唯一需要 NAT 的场景是 Lambda 主动向外发起连接（比如调 S3、调外部 API）。我的 Lambda 只做一件事——连 PostgreSQL 返回结果，不调任何外部服务，所以完全不需要 NAT，零额外费用。
>
> ### Q5: "How do you handle the 38M row table — won't the query be slow?"
>
> > 三层过滤。第一，`WHERE status != 7` 只查失败记录，排除了绝大多数成功的 job。第二，`AND jobsubmittime >= NOW() - INTERVAL '60 days'` 加了时间窗口，避免全表扫描。第三，`ORDER BY jobsubmittime DESC` 配合 60 天窗口，返回量通常在几十到几百条。Lambda 实测延迟 172ms，完全没有性能问题。如果未来数据量增长，可以在 `jobsubmittime` + `status` 上加复合索引，或者缩短时间窗口。
>
> ### Q6: "Why REST API Gateway instead of AppSync for Tab B?"
>
> > Tab A 用 AppSync 是因为需要 GraphQL subscription（WebSocket 实时推送）。Tab B 是简单的 GET 请求 + 轮询，不需要 subscription，用 REST API Gateway 最直接——一个 GET endpoint，Lambda proxy integration，5 分钟就能配好。如果用 AppSync，还要定义 schema、写 resolver、处理 IAM auth，对于一个简单的只读轮询来说是过度设计。
>
> ### Q7: "How does authentication work for Tab B?"
>
> > Tab B 不需要单独的后端认证。整个 dashboard 被 Cognito `<Authenticator>` 组件包裹——未登录用户看到的是登录页面，根本执行不到 Tab B 的代码（包括发起 API 请求的 `useEffect`）。API Gateway 的 CORS 设为 `*` 允许任何域名调用，但实际上只有已登录用户的浏览器才会发起请求。如果将来需要更严格，可以给 API Gateway 加 Cognito Authorizer 或 IAM auth。
>
> ### Q8: "What happens when a failed job gets retried and succeeds?"
>
> > 这是轮询方案最优雅的地方——什么都不用做。每 30 秒前端拉一次全量数据，Lambda 的 SQL 查的是 `status != 7`（非成功）。当一个 job 重试成功后，它的 status 变成 7，下次查询自然就不包含它了。前端做的是 full list replacement，不是增量更新，所以"修复了的记录自动消失"是免费得到的行为，不需要任何删除逻辑。
>
> ------
>

>
> ```
> Tab A (existing, real-time):
> NiFi → DynamoDB → DynamoDB Streams → Lambda → AppSync Mutation
>   (NONE resolver) → Subscription (WebSocket) → React
> 
> Tab B (new, polling):
> PostgreSQL (EC2, PgBouncer:6432)

>     │

## EC2 Auto Start/Stop 

> ------
>
> ## 一句话 Elevator Pitch
>
> > "I built a serverless scheduling system on AWS that automatically stops and starts EC2 instances on a daily office-hours cadence, scoped by resource tags for safety, saving the team unnecessary compute costs with zero manual intervention."
>
> ------
>
> ## STAR 框架回答
>
> ### Situation
>
> - 团队有多台 EC2 实例只在工作时间使用（dev/staging 环境），但经常忘记关机，导致非工作时间产生不必要的 compute 费用
> - 实例分布在 **us-east-1** 和 **us-east-2** 两个 region
>
> ### Task
>
> - 设计一个自动化方案：每天 **6 AM start、6 PM stop**（Eastern Time）
> - 必须安全——不能误伤不该关的实例
> - 必须支持挂载了 **KMS 加密 EBS 卷**的实例
> - 新增实例时零配置变更（只打 tag 就行）
>
> ### Action
>
> **架构选型：EventBridge Scheduler → SSM Automation**
>
> 我没有用 Lambda，而是选了 fully managed 的方案：EventBridge Scheduler 做定时触发，SSM Automation 的官方 runbook（`AWS-StopEC2Instance` / `AWS-StartEC2Instance`）做执行。这样整条链路没有自定义代码，维护成本最低。
>
> **安全设计：Tag-based scoping + least-privilege IAM**
>
> - 定义约定 tag `AutoStartStop=OfficeHours`，只有打了这个 tag 的实例才会被操作
> - IAM policy 用 `aws:ResourceTag` condition 做强制约束——即使 runbook 被错误触发，没有 tag 的实例也不会被操作
> - 整个方案涉及两个 IAM role，职责分离：
>   - **Scheduler 执行角色**：只有 `ssm:StartAutomationExecution` + `iam:PassRole`
>   - **SSM Automation 执行角色**：只有 `ec2:Start/StopInstances`（带 tag condition）+ 必要的 KMS 权限
>
> **多 Region 扩展**
>
> - IAM role 是全局的，直接复用
> - Policy 里追加新 region 的 resource ARN
> - 每个 region 各建一套 EventBridge Schedule（因为 Scheduler 和 SSM 都是区域服务，天然隔离）
>
> **踩坑与排障**
>
> 在冒烟测试阶段发现并解决了三个问题：
>
> | 问题                                     | 根因                                                | 修复                                        |
> | ---------------------------------------- | --------------------------------------------------- | ------------------------------------------- |
> | Stop 轮询阶段 403                        | 缺 `ec2:DescribeInstanceStatus`                     | 加到 policy                                 |
> | Start 超时，报 `InternalError on launch` | 加密 EBS 卷需要 `kms:CreateGrant`                   | 给 SSM role 加 KMS 权限                     |
> | Scheduler retry 不触发                   | Retry 只针对 API 调用失败，不管 Automation 内部超时 | 加长 retention time + 考虑延迟补刀 schedule |
>
> ### Result
>
> - 非工作时间 EC2 compute 费用降为零（对覆盖的实例）
> - 新增实例只需打一个 tag，无需改 IAM 或 schedule 配置
> - 方案已扩展到两个 region，互不干扰
>
> ------
>
> ## 高频追问 & 回答要点
>
> ### Q1: "Why SSM Automation instead of Lambda?"
>
> > SSM 官方 runbook 已经封装了 stop/start + 状态轮询逻辑，不需要写代码。Lambda 方案需要自己写轮询、处理超时、打日志，维护成本更高。选 managed service 能做的事就不写代码。
>
> ### Q2: "Why two IAM roles?"
>
> > Separation of concerns。Scheduler 的角色只需要"发起 automation"的权限；SSM Automation 的角色才需要"操作 EC2"的权限。如果合成一个，Scheduler 就间接拥有了 EC2 操作权限，违反 least privilege。中间用 `iam:PassRole` 做显式的权限传递。
>
> ### Q3: "How does tag-based scoping work as a safety mechanism?"
>
> > 两层防护。第一层：SSM Automation 的 `Targets` 参数用 tag 做查询，只会找到有 tag 的实例。第二层：即使绕过了第一层，IAM policy 的 `aws:ResourceTag` condition 会在 API 层面拒绝对没有 tag 的实例的操作。Defense in depth。
>
> ### Q4: "Multi-region — won't they interfere with each other?"
>
> > 不会。三道区域隔离：
> >
> > 1. **Scheduler 是区域服务** — 只调用本 region 的 SSM endpoint
> > 2. **SSM 是区域服务** — 展开 tag 时只查本 region 的实例
> > 3. **EC2 是区域服务** — API 只发给本 region 的 endpoint
> >
> > Policy 里写了多个 region 的 ARN 只是"两边都放行"，不会让一个 region 的请求跑去操作另一个 region。**Policy 是门卫，不是司机。**
>
> ### Q5: "What was the hardest debugging moment?"
>
> > Start 超时那个坑。现象是 Stop 每次成功，Start 偶尔失败，报 `Server.InternalError: Internal error on launch`。手动从 Console Start 没问题。排查发现是加密 EBS 卷的问题——EC2 启动时需要 `kms:CreateGrant` 来解密卷，SSM 的 role 没有这个权限。手动操作没问题是因为我自己的 IAM user 有 KMS 权限。修复后确认 KMS key policy 里有默认的 IAM delegation（`root` principal + `kms:*`），所以只需要改 IAM policy，不用动 key policy。
>
> ### Q6: "What happens if someone manually starts an instance that the scheduler will stop?"
>
> > 如果实例有 tag，它会按计划被 stop——这是预期行为。如果不想被自动管理，不打 tag 就行。Opt-in 模型比 opt-out 更安全。
>
> ### Q7: "How would you monitor this in production?"
>
> > 几个维度：
> >
> > - **EventBridge Scheduler** 有 invocation metrics（成功/失败），接 CloudWatch Alarm
> > - **SSM Automation** 每次执行有 status（Success/Failed/TimedOut），可以用 EventBridge rule 监听 `SSM Automation Execution Status Change` 事件，失败时发 SNS 通知
> > - 也可以加一个 cost anomaly detection 做兜底——如果某天非工作时间费用异常，说明 schedule 可能没生效
>
> ------
>
> ## 画架构时的关键词
>
> ```
> EventBridge Scheduler (per region)
>         │
>         │  assume → Scheduler Execution Role
>         │  call   → ssm:StartAutomationExecution
>         │  pass   → SSM Automation Role (via iam:PassRole)
>         ▼
> SSM Automation (per region)
>         │
>         │  assume → SSM Automation Execution Role
>         │  query  → tag:GetResources (find tagged instances)
>         │  call   → ec2:StopInstances / ec2:StartInstances
>         │  call   → kms:CreateGrant (for encrypted EBS, start only)
>         ▼
> EC2 Instances (tagged: AutoStartStop=OfficeHours)
> ```
>
> ------
>
> ## 关键术语速查
>
> | 术语                        | 一句话解释                                               |
> | --------------------------- | -------------------------------------------------------- |
> | EventBridge Scheduler       | Fully managed cron service，支持时区                     |
> | SSM Automation              | 用 runbook 编排 AWS API 调用，无需 EC2 agent             |
> | `iam:PassRole`              | 允许一个 principal 把某个 role "交给"另一个 service 使用 |
> | `aws:ResourceTag` condition | IAM policy 里按目标资源的 tag 做访问控制                 |
> | KMS Grant                   | 临时授权机制，EC2 启动时用来解密加密 EBS 卷              |
> | Defense in depth            | 多层安全防护，任一层失效其他层仍能保护                   |

## ECS Fargate Auto Start/Stop

> ------
>
> ## 一句话 Elevator Pitch
>
> > "I built a tag-driven serverless system that automatically scales 30+ ECS Fargate services to zero every evening and restores them every morning, across multiple regions — no per-service configuration needed, just tag and forget."
>
> ------
>
> ## STAR 框架回答
>
> ### Situation
>
> - 团队有 30+ ECS Fargate services 跑在 dev/staging 环境（分布在 us-east-1 和 us-east-2）
> - 这些 services 只在工作时间使用，但 24/7 运行，非工作时间白白烧钱
> - 和 EC2 不同，ECS 没有原生的"stop instance"概念——Fargate 按 task 计费，唯一的"关机"方式是把 `desiredCount` 设为 0
>
> ### Task
>
> - 每天 6 PM stop、6 AM start（Eastern Time），覆盖所有 tagged services
> - **不能逐个 service 配置**——30+ services 跨多个 cluster，手动维护不可接受
> - 新增 service 只需打 tag，零配置变更
> - 必须支持多 region，互不干扰
>
> ### Action
>
> **架构选型：EventBridge Scheduler → Lambda → ECS UpdateService**
>
> 我评估了四种方案后选了这个：
>
> | 方案                                 | 否决原因                                                     |
> | ------------------------------------ | ------------------------------------------------------------ |
> | EventBridge 直接调 ECS UpdateService | 每个 service 需要一对独立 schedule，30+ services = 60+ schedules，不可维护 |
> | SSM Automation                       | runbook 必须硬编码所有 cluster/service 名，且要在 JSON 里写 Python，更难维护 |
> | Step Functions (no Lambda)           | ASL 定义冗长，tag 解析需要 JSONata，本质上是"用 JSON 写 Python"，可读性更差 |
> | **Lambda + Scheduler** ✅             | 一个 Lambda 按 tag 动态扫描所有 services，新增 service 只打 tag，O(1) 维护成本 |
>
> **核心设计：Tag-driven discovery**
>
> 定义两个标准 tag：
>
> - `Ecs-Service-AutoSchedule = OfficeHours`（opt-in 标记）
> - `DesiredCount = N`（记录恢复时的目标值，每个 service 可以不同）
>
> Lambda 用 Resource Groups Tagging API (`tag:GetResources`) 动态扫描所有带 tag 的 `ecs:service` 资源，从返回的 long ARN 里解析出 cluster name 和 service name，然后调 `ecs:UpdateService`。
>
> Stop 时 `desiredCount=0`，Start 时从 tag 读 `DesiredCount` 值恢复。
>
> **IAM 安全设计：三层防护**
>
> 1. **Tag condition**：`ecs:UpdateService` 的 IAM policy 用 `aws:ResourceTag` condition，只允许操作带 tag 的 service
> 2. **Region lock**：每个 region 的 Lambda role 有显式 Deny（`aws:RequestedRegion != 当前region`），即使代码出 bug 也不会跨 region 操作
> 3. **Scheduler role 最小权限**：只有 `lambda:InvokeFunction` 权限，且 resource 锁定到具体函数 ARN
>
> **多 Region 架构**
>
> 每个 region 独立一套：
>
> | 组件                     | 是否跨 region 复用                             |
> | ------------------------ | ---------------------------------------------- |
> | Tag key/value            | ✅ 全局统一                                     |
> | Lambda 代码              | ✅ 完全相同（boto3 自动用当前 region endpoint） |
> | Lambda function          | ❌ 每个 region 各一个                           |
> | Lambda IAM role + policy | ❌ 每个 region 各一个（ARN 锁定到该 region）    |
> | Scheduler role           | ❌ 每个 region 各一个（invoke ARN 不同）        |
> | Schedules (stop + start) | ❌ 每个 region 各一对                           |
>
> ### Result
>
> - 非工作时间 Fargate 费用降为零（对覆盖的 services）
> - 30+ services 的管理成本 = 打两个 tag，无需改任何 schedule 或 IAM
> - 新增 service / 新增 region 有标准化 playbook，10 分钟可铺开
> - 方案已在 us-east-1 和 us-east-2 稳定运行
>
> ------
>
> ## 高频追问 & 回答要点
>
> ### Q1: "Why not just use EventBridge Scheduler to call ECS UpdateService directly?"
>
> > EventBridge Scheduler 的 Universal Target 可以直接调 `ecs:UpdateService`，但它是一对一的——每个 service 需要一对 stop/start schedule。我们有 30+ services 跨多个 cluster，这意味着 60+ schedules，每新增一个 service 还要手动建两个 schedule。Lambda 方案只需要一对 schedule（stop + start），Lambda 自己按 tag 动态发现所有目标，维护成本是 O(1) vs O(n)。
>
> ### Q2: "Why is this harder than EC2 auto start/stop?"
>
> > EC2 实例是"flat"的——所有实例在一个 region 的池子里，ID 全局唯一，SSM Automation 的 `Targets` 参数原生支持 tag-based targeting。但 ECS 有 cluster 隔离——同名 service 在不同 cluster 里是完全不同的资源，而且 ECS 没有原生的 "stop service" API，只能把 `desiredCount` 设为 0。所以我需要一个中间层（Lambda）来做 tag 扫描 → ARN 解析 → 跨 cluster 批量操作。
>
> ### Q3: "How does the Lambda discover services across multiple clusters?"
>
> > 不是按 cluster 遍历的。我用的是 Resource Groups Tagging API (`tag:GetResources`)，它是 region 级的，一次调用就能返回该 region 内所有带指定 tag 的 `ecs:service` 资源。返回的是 long ARN 格式：`arn:aws:ecs:region:account:service/cluster-name/service-name`，我直接 parse 出 cluster 和 service name，不需要事先知道有哪些 cluster。
>
> ### Q4: "How do you handle different services needing different desiredCount?"
>
> > `desiredCount` 存在 tag 里（`DesiredCount=N`），不是硬编码在 Lambda 或 schedule 里。Stop 时统一设 0，Start 时从每个 service 自己的 tag 读值。如果一个 service 平时跑 3 个 task，它的 tag 就是 `DesiredCount=3`。这样每个 service 可以独立控制恢复值，Lambda 代码完全通用。
>
> ### Q5: "Walk me through the IAM design."
>
> > 整个方案有三类 role，职责完全分离：
> >
> > **1) Lambda execution role**（每 region 一个）
> >
> > - `tag:GetResources`：用来发现 tagged services
> > - `ecs:UpdateService`：带 `aws:ResourceTag` condition，只能操作有 tag 的 service
> > - 显式 Deny：`aws:RequestedRegion != 当前region`，硬性阻止跨 region 操作
> >
> > **2) Scheduler execution role**（每 region 一个）
> >
> > - 只有 `lambda:InvokeFunction`，resource 锁定到具体函数 ARN
> >
> > **3) ECS task execution role**（已有的 `ecsTaskExecutionRole`）
> >
> > - 这个不是我创建的，是 ECS 拉镜像和写日志用的，和 scheduling 无关
> >
> > 关键设计思想：Scheduler 只能触发 Lambda，Lambda 只能操作 tagged services in current region。任何一层被误配，其他层仍然保护。
>
> ### Q6: "What if the Lambda fails mid-execution — some services stopped, some didn't?"
>
> > Lambda 代码是 best-effort 循环：对每个 service 的 `UpdateService` 调用独立 try-catch，失败的记录到 errors 数组但不中断其他 service。返回值里有 `updatedCount` 和 `errorCount`，方便排查。而且 stop 和 start 都是幂等的——已经是 0 的再设 0 没有副作用，已经 running 的再设 `desiredCount=1` 也没有副作用。Scheduler 配了 retry（3 次 / 6 小时 retention），如果 Lambda 本身超时，Scheduler 会重试。
>
> ### Q7: "How would you handle multiple time zones in the same region?"
>
> > 目前的设计是一个 region 对应一个作息时区。如果同一个 region 需要多个时区（比如 us-east-1 里既有纽约团队也有伦敦团队的 services），就需要再加一个分组 tag（比如 `ScheduleGroup=NYC` / `ScheduleGroup=LDN`），Lambda 的 event payload 里带 group，按 group 过滤。但实际上大多数公司按 region 分时区就够了。
>
> ### Q8: "EC2 项目和 ECS 项目的对比？"
>
> > | 维度        | EC2 方案                               | ECS 方案                           |
> > | ----------- | -------------------------------------- | ---------------------------------- |
> > | 触发层      | EventBridge Scheduler                  | EventBridge Scheduler              |
> > | 执行层      | SSM Automation（官方 runbook，零代码） | Lambda（~60 行 Python）            |
> > | Target 发现 | SSM 原生支持 `Targets: tag:`           | 需要自己调 Tagging API             |
> > | "关机"机制  | `ec2:StopInstances`                    | `ecs:UpdateService` desiredCount=0 |
> > | 额外复杂度  | KMS 权限（加密 EBS）                   | desiredCount 恢复值存 tag          |
> > | 代码量      | 零                                     | ~60 行 Python                      |
> >
> > 选 SSM 还是 Lambda 取决于 AWS 是否已经提供了你需要的 managed 抽象。EC2 有，ECS 没有，所以 ECS 需要一层薄薄的胶水代码。
>

> ## 画架构时的关键词
>
> ```
> EventBridge Scheduler (per region, timezone-aware)
>         │
>         │  assume → Scheduler Invoke Role
>         │  call   → lambda:InvokeFunction
>         │  payload: {"action": "stop"} or {"action": "start"}
>         ▼
> Lambda (per region, ~60 lines Python)
>         │
>         │  1. tag:GetResources → find all ecs:service with tag
>         │  2. Parse long ARN → extract cluster + service name
>         │  3. ecs:UpdateService → set desiredCount
>         │     stop: desiredCount = 0
>         │     start: desiredCount = tag[DesiredCount]
>         ▼
> ECS Fargate Services (tagged: Ecs-Service-AutoSchedule=OfficeHours)
>   ├── Cluster A: svc-1, svc-2, svc-3 ...
>   └── Cluster B: svc-4, svc-5 ...
> ```
>

> ## 关键术语速查
>
> | 术语                        | 一句话解释                                               |
> | --------------------------- | -------------------------------------------------------- |
> | ECS Fargate                 | Serverless 容器运行环境，按 task 计费，没有底层 EC2 可管 |
> | desiredCount                | ECS service 的目标 task 数量，设为 0 等于"关机"          |
> | Resource Groups Tagging API | 跨服务的 tag 查询 API，一次调用扫描整个 region           |
> | Long ARN                    | ECS 新格式 ARN，包含 cluster/service 路径，支持 tagging  |
> | `aws:RequestedRegion`       | IAM condition key，限制 API 调用只能发往指定 region      |
> | `aws:ResourceTag`           | IAM condition key，按目标资源的 tag 做访问控制           |
> | 幂等性                      | 重复调用结果不变（已 stopped 再 stop 无副作用）          |
